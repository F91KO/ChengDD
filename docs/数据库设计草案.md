# 程哒哒数据库设计草案

## 1. 文档说明

- 文档用途：基于当前 PRD、服务边界和服务与表关系矩阵，输出第一版数据库设计草案。
- 当前阶段：逻辑数据库设计，不等同于最终建表 SQL。
- 适用对象：产品、后端研发、测试、架构、实施。

## 2. 数据库设计原则

- 数据库类型：`MySQL 5.7`
- 字符集：`utf8mb4`
- 表命名统一采用：`cdd_{domain}_{object}`
- 一期采用单库多表，逻辑按服务边界分域
- 所有核心业务表统一保留租户边界字段
- 跨服务不直接写其他服务主表
- 配置类、模块类、扩展类复杂字段可优先使用 `json`
- 审计日志、状态日志、埋点汇总优先做追加写，避免频繁更新大字段
- 一期数据库设计不依赖 MySQL 8 专属特性，便于低成本私有化交付。

## 3. 通用字段规范

### 3.1 推荐公共字段

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | bigint | 主键 ID |
| `merchant_id` | bigint | 商家 ID，平台级共享表可为空 |
| `store_id` | bigint | 店铺 ID |
| `mini_program_id` | bigint | 小程序 ID |
| `created_by` | bigint | 创建人 |
| `updated_by` | bigint | 更新人 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |
| `deleted` | tinyint(1) | 逻辑删除标记，`0/1` |
| `version` | bigint | 乐观锁版本号 |

### 3.2 主键建议

- 主键统一采用 `bigint`
- 不使用数据库自增主键
- 主键生成方式建议采用应用侧发号，优先使用号段模式
- 号段模式可按业务域分配，例如商家、商品、订单、发布任务分别维护独立号段
- 对外暴露时可继续保留业务编码字段，如 `order_no`、`task_no`

### 3.2.1 主键生成建议

| 方案 | 是否推荐 | 说明 |
| --- | --- | --- |
| 数据库自增 | 否 | 不满足当前主键策略 |
| 雪花算法 | 备选 | 全局唯一，但可读性较差 |
| 号段模式 | 是 | 兼顾 `bigint` 主键、全局唯一和可控发号 |

推荐做法：
- 增加统一发号表，例如 `cdd_config_id_segment`
- 字段建议包括：`biz_code`、`current_max_id`、`step_size`、`version`
- 应用启动后按业务域预取号段到内存，减少频繁访问数据库
- 核心业务编号仍独立设计，例如 `order_no`、`refund_no`、`task_no`

### 3.3 索引建议

- 所有主查询条件必须有索引
- 所有外键语义字段至少考虑普通索引
- 高频联合查询优先设计联合索引
- JSON 字段不作为核心过滤条件，必要时拆出冗余字段

- `id`
- `biz_code`
- `current_max_id`
- `step_size`
- `version`
- `status`
- `updated_at`

字段说明：
- `biz_code`：业务编码，例如 `merchant`、`store`、`product`、`order`、`release_task`
- `current_max_id`：当前已经分配到的最大 ID
- `step_size`：每次预分配的号段大小，例如 `100`、`500`、`1000`
- `version`：乐观锁版本号，用于并发发号控制

索引建议：
- `uk_biz_code`

## 13. 业务单号规则建议

### 13.1 设计原则

- 主键 `id` 与业务单号分离设计
- 主键负责内部关联，业务单号负责展示、检索、排查和客户沟通
- 业务单号应具备可读性，但不要求参与主键关联
- 业务单号建议按“业务前缀 + 日期 + 顺序号”生成

### 13.2 推荐单号字段

| 对象 | 字段 | 示例 |
| --- | --- | --- |
| 商家 | `merchant_no` | `MER202603200001` |
| 店铺 | `store_no` | `ST202603200001` |
| 订单 | `order_no` | `OD20260320000123` |
| 支付流水 | `pay_no` | `PY20260320000056` |
| 退款流水 | `refund_no` | `RF20260320000018` |
| 售后单 | `after_sale_no` | `AS20260320000009` |
| 发布任务 | `task_no` | `RL20260320000011` |

### 13.3 推荐业务前缀

| 对象 | 前缀 |
| --- | --- |
| 商家 | `MER` |
| 店铺 | `ST` |
| 订单 | `OD` |
| 支付 | `PY` |
| 退款 | `RF` |
| 售后 | `AS` |
| 发布任务 | `RL` |

### 13.4 生成建议

- 日期部分建议采用 `yyyyMMdd`
- 顺序号建议按日重置或按全局递增二选一
- 若客户更看重排查便利，建议按日重置，例如 `OD20260320000123`
- 若客户更看重实现简单，建议全局递增后左补零
- 单号生成可基于独立序列表或 Redis 计数器，但最终落库前需校验唯一性
