## 2. 项目背景

建设一个支持多商户发布小程序的平台型系统，平台定位是帮助商家发布小程序、提供商城通用功能，并以尽可能低成本的方式满足部署和交付要求。平台方负责商户接入、模板提供、账号权限、分类模板、版本治理和发布能力，商户负责各自的小程序经营内容与业务运营。

前台采用“同一套代码 + 多商家独立小程序”的模式：平台维护统一的小程序代码基座，不同商家可绑定各自的小程序 AppID，基于统一代码通过配置、功能开关和扩展点能力生成各自的小程序版本。消费者进入某个商家的小程序后，仅浏览、购买并评价该商家的商品，不展示其他商家数据。

本期目标不是为每个商家单独开发一套前端代码，而是建设统一模板、统一发布、统一治理、提供商城通用功能、支持定制化配置和有限扩展的商家小程序 SaaS 能力。

## 3. 项目目标

### 3.1 业务目标

1. 支持同一套代码服务多个商户的独立小程序发布场景。
2. 支持商户按自身品牌和经营模式进行定制化配置。
3. 支持平台统一管理模板、分类模板、账号权限和版本发布。
4. 支持面向平台方或甲方的源码交付与私有化部署。
5. 支持在满足要求的前提下优先采用低成本部署方案，并可提供 CI/CD 服务。

### 3.2 产品目标

1. 构建消费者小程序、商家后台、平台交付后台三类核心工作台，并提供商城通用功能能力。
2. 打通商户注册、小程序接入、店铺配置、装修配置、商品维护和版本发布全链路。
3. 建立标准配置、功能开关、审批制扩展三层定制化能力。
4. 建立多商家、多店铺、多子账号场景下的版本管理、租户隔离、消息通知、风控审计能力。
5. 建立可交付、可部署、可二次开发的源码包管理能力。

### 3.3 技术目标

1. 所有商家小程序基于统一代码主干发布，不允许为单商家长期维护独立代码分支。
2. 商家差异优先通过配置化和组件化实现，减少重复开发。
3. 版本发布支持按商家、按店铺/小程序灰度、回滚、兼容性校验和审计留痕。

### 3.4 成功指标

1. 商户完成基础信息配置后 1 个工作日内可完成小程序接入和首版发布。
2. 满足前置条件的标准商户可在 30 分钟内完成一键开通和一键首版发布。
3. 80% 以上商家定制需求可通过标准配置或功能开关完成。
4. 平台配置、发布、账号和商户数据按商户维度严格隔离。
5. 单次模板版本发布支持不少于 50 个商家小程序复用。

### 3.5 平台方目标

1. 通过统一代码、统一模板、统一发布机制降低多商家交付成本。
2. 通过分类模板、标准配置、功能开关提升商家开通效率，减少重复实施工作。
3. 通过多商家隔离、发布治理、账务对账、日志审计提升平台可管可控能力。
4. 通过源码交付和私有化部署能力满足甲方项目制交付场景。
5. 在不为每个商家长期维护独立分支的前提下，支持必要的定制化诉求。
6. 在满足项目要求的前提下，优先采用低成本部署和标准化交付方式，并可按需提供 CI/CD 服务。

#### 3.5.1 平台方核心诉求拆解

| 目标 | 说明 | 需要落地的能力 |
| --- | --- | --- |
| 降低交付成本 | 多商家复用同一套代码和模板 | 模板管理、统一发布、配置中心 |
| 缩短交付周期 | 商家尽量通过模板和配置快速上线 | 一键开通、模板初始化、默认配置 |
| 保持平台治理能力 | 商家可定制但不能失控 | 功能开关、扩展点审批、审计日志 |
| 保障平台稳定性 | 一个商家的变更不能影响其他商家 | 租户隔离、灰度发布、回滚机制 |
| 保证项目可售可交付 | 面向客户可承诺源码和私有化能力 | 源码交付包、部署手册、版本映射 |

#### 3.5.2 平台方面临的典型问题与对应方案

| 典型问题 | 影响 | 本方案对应解法 |
| --- | --- | --- |
| 每接一个商家就要重复开发或复制项目 | 研发与实施成本高，版本难维护 | 统一代码主干 + 模板化交付 |
| 商家差异需求多，容易形成失控分支 | 长期版本不可收敛 | 配置优先、开关治理、审批制扩展 |
| 商家上线依赖人工操作过多 | 项目启动慢，交付周期不可控 | 一键开通、一键发布、标准校验流程 |
| 多商家并行发布时风险大 | 容易互相影响 | 商家级灰度发布、版本映射、回滚机制 |
| 私有化项目交付难标准化 | 交付内容不一致，售后成本高 | 标准源码包、部署手册、初始化脚本 |

#### 3.5.3 平台方收益路径

| 阶段 | 预期收益 | 衡量方式 |
| --- | --- | --- |
| 初期 | 能快速承接多商家项目，缩短首单交付周期 | 从签约到首版上线周期 |
| 中期 | 高复用率降低交付和维护成本 | 模板复用率、通用能力复用率 |
| 后期 | 平台形成稳定的标准产品与交付体系 | 标准化交付占比、定制回收率 |

### 3.6 商家目标

1. 低门槛入驻并快速开通自己的小程序商城。
2. 在统一平台能力上保留自己的品牌表达、分类结构和经营方式。
3. 无需理解复杂技术细节即可完成商品上架、分类维护、页面装修和发布。
4. 在不被平台强制统一商品分类的情况下，使用自己的分类树经营商品。
5. 获得清晰、可追溯的账单与退款对账能力，商家收入不计入平台。

#### 3.6.1 商家核心诉求拆解

| 目标 | 说明 | 需要落地的能力 |
| --- | --- | --- |
| 快速上线 | 少填资料、少改代码、少走人工流程 | 一键开通、一键发布、默认模板 |
| 自主经营 | 店铺风格、分类、营销可自行调整 | 分类模板复制、装修配置、功能开关 |
| 低学习成本 | 后台操作应接近标准 SaaS | 分类管理页、商品管理页、可视化配置 |
| 可持续迭代 | 上线后可逐步调整能力 | 配置发布、开关治理、局部扩展 |
| 账务清晰 | 看得懂订单、退款和账单变化 | 账单中心、对账中心、账务明细 |

#### 3.6.2 商家在不同阶段的核心关注点

| 阶段 | 商家关心的问题 | 系统需要提供的支持 |
| --- | --- | --- |
| 入驻前 | 能否快速通过审核，资料要准备什么 | 入驻指引、材料清单、进度查询 |
| 开通期 | 能否尽快完成接入并生成可用商城 | 一键开通、模板初始化、自动校验 |
| 上线初期 | 商品怎么分类、怎么装修、怎么发版 | 分类管理、装修中心、发布结果反馈 |
| 稳定经营期 | 如何提升转化和运营效率 | 功能开关、数据看板、营销能力 |
| 扩展期 | 特殊经营模式能否被支持 | 扩展点机制、定制申请、版本兼容评估 |

#### 3.6.3 商家实际获得的价值

1. 不需要从零搭建小程序技术体系，即可拥有独立商城。
2. 可以使用平台提供的模板快速起步，但保留自己的品牌和分类结构。
3. 能以可视化方式完成分类、商品、页面和功能管理，降低运营门槛。
4. 经营过程中若产生新需求，可先尝试配置与开关，再进入扩展申请，不会一开始就陷入重开发。
5. 账单和退款对账链路清晰，方便日常运营和财务核对。

### 3.7 平台方实现路径

#### 3.7.1 第一阶段：建设统一底座

1. 建立统一小程序代码基座、统一后端服务和统一配置中心。
2. 建立商家、店铺、小程序接入、商品、订单、账单等核心数据模型。
3. 建立租户识别、权限隔离、日志审计和版本管理能力。

#### 3.7.2 第二阶段：建设标准化交付能力

1. 建立商家入驻、审核、接入、一键开通、一键发布流程。
2. 建立平台模板体系，包括页面模板、分类模板、功能模板。
3. 建立默认配置初始化机制，保证标准商家可直接上线。

#### 3.7.3 第三阶段：建设可配置经营能力

1. 建立商家装修中心、商品分类管理、商品管理、订单履约、售后处理能力。
2. 建立功能开关中心，让商家在授权范围内启停营销和交易能力。
3. 建立账单与对账能力，支持退款冲减和账期管理。

#### 3.7.4 第四阶段：建设可治理扩展能力

1. 建立扩展点注册机制和审批制扩展流程。
2. 建立模板版本治理、商家版本映射、灰度发布和回滚能力。
3. 建立模板偏离分析、商家分类偏离分析和商家能力使用分析。

#### 3.7.5 第五阶段：建设交付与复制能力

1. 输出标准源码交付包、部署文档、初始化脚本和版本说明。
2. 建立私有化部署适配能力，支持项目制交付。
3. 将项目中高频定制需求回收为平台标准能力，持续提高复用率。

#### 3.7.6 平台方实施任务拆解

| 领域 | 重点任务 | 产出物 |
| --- | --- | --- |
| 产品 | 明确模板边界、开关边界、扩展边界、分类模板规则 | PRD、流程图、原型、规则说明 |
| 研发 | 建设统一代码、配置中心、版本中心、扩展机制 | 可运行系统、接口、数据模型 |
| 测试 | 验证多商家隔离、模板兼容、发布回滚、账务准确性 | 测试用例、回归报告、验收记录 |
| 运营 | 维护模板、审核商家、治理分类和内容 | 运营手册、模板库、审核标准 |
| 实施交付 | 输出源码包、部署脚本、初始化方案 | 交付清单、部署手册、环境配置模板 |

#### 3.7.7 平台方关键控制点

1. 所有商家必须先走模板和配置路径，只有确有必要时才进入审批制扩展。
2. 商家发布前必须经过接入校验、配置校验、分类校验、商品校验和版本兼容校验。
3. 高风险能力必须由平台强控，例如支付、账务、隐私和受监管类商品能力。
4. 私有化项目交付后仍需保留主干回流机制，避免长期偏离平台标准产品。

#### 3.7.8 平台方阶段交付建议

| 阶段 | 重点交付内容 | 是否面向客户可见 |
| --- | --- | --- |
| 统一底座阶段 | 租户模型、统一代码、配置中心、版本中心 | 否 |
| 标准化交付阶段 | 入驻、一键开通、模板初始化、基础模板 | 是 |
| 可配置经营阶段 | 分类管理、装修中心、商品与订单能力、账单中心 | 是 |
| 可治理扩展阶段 | 功能开关、扩展点、审批流、灰度与回滚 | 部分可见 |
| 交付复制阶段 | 源码交付包、部署手册、初始化脚本 | 是 |

### 3.8 商家落地路径

#### 3.8.1 开通前

1. 商家提交主体资料并完成审核。
2. 商家完成小程序 AppID、支付参数、基础协议确认。
3. 商家选择一个平台分类模板和页面模板作为初始方案。
4. 商家主账号可预先规划店铺与子账号的对应关系。

#### 3.8.2 开通中

1. 商家通过一键开通页补全店铺信息、主题配置、客服信息。
2. 系统自动复制分类模板，初始化商家自己的分类树。
3. 系统自动校验接入参数、模板兼容性、商品数量和必填配置。
4. 若商家存在多个小程序，系统应支持分别选择目标店铺/小程序进行初始化和发布。

#### 3.8.3 开通后首期运营

1. 商家维护自己的分类树和商品归类关系。
2. 商家完成首页装修、商品上架、基础营销配置。
3. 商家执行首版发布并开始接单、发货、售后处理。
4. 商家主账号可将不同店铺/小程序分别授权给不同子账号运营维护。

#### 3.8.4 经营中迭代

1. 商家可逐步调整分类结构、页面装修和功能开关。
2. 商家可根据经营情况申请审批制扩展。
3. 商家通过后台持续查看访客、订单、退款和账单数据。

#### 3.8.5 商家操作路径细化

| 阶段 | 商家动作 | 系统响应 |
| --- | --- | --- |
| 选择模板 | 选择页面模板和分类模板 | 初始化默认配置和商家分类树 |
| 配置基础信息 | 填写店铺、客服、主题色、发货时效 | 保存配置草稿并生成校验任务 |
| 上架商品 | 创建商品并挂载到商家分类 | 自动校验分类是否有效、资质是否满足 |
| 首版发布 | 点击一键发布 | 生成发布任务并返回发布结果 |
| 日常经营 | 调整分类、装修页面、启停功能开关、分配子账号权限 | 版本化保存配置并记录审计日志 |
| 特殊扩展 | 提交定制需求 | 进入审批、评估、开发、灰度流程 |

#### 3.8.6 商家成功上线的前置条件

1. 主体审核已通过。
2. 小程序接入配置已验证通过。
3. 至少选择一个页面模板和一个分类模板，或经平台允许使用空白分类树。
4. 至少存在约定数量的已上架商品。
5. 必填协议、客服信息、发货时效和售后规则已完成配置。

### 3.9 平台方与商家对照说明

#### 3.9.1 双方目标对照

| 维度 | 平台方目标 | 商家目标 |
| --- | --- | --- |
| 系统建设 | 建一个可复制、可治理的平台 | 用最短时间获得可用商城 |
| 分类能力 | 提供模板并控制边界 | 维护自己的分类树 |
| 版本机制 | 统一主干、统一发布、统一回滚 | 发布简单、变更可追溯 |
| 定制方式 | 控制定制成本和风险 | 满足自身经营差异 |
| 账务能力 | 保证账单清晰和对账准确 | 看懂收入、退款和账单变化 |

#### 3.9.2 双方职责分工

| 事项 | 平台方负责 | 商家负责 |
| --- | --- | --- |
| 入驻审核 | 审核主体、协议、资质 | 提交真实完整资料 |
| 小程序接入 | 提供接入规范和校验能力 | 提供 AppID、密钥、支付参数 |
| 分类体系 | 提供分类模板和规则边界 | 维护自己的分类树 |
| 页面与功能 | 提供模板、开关、扩展机制 | 完成装修配置和经营启停 |
| 账号权限 | 提供账号模型、RBAC 和数据权限机制 | 创建子账号并分配店铺/小程序权限 |
| 版本发布 | 提供发布机制、灰度、回滚 | 发起发布并处理配置问题 |
| 账务处理 | 提供账单与退款对账能力 | 核对账单和退款记录 |

#### 3.9.3 系统落地原则

1. 平台提供的是标准化经营底座，不是完全自由的定制开发平台。
2. 商家拥有经营自主权，但不能突破平台设定的安全、资质和版本治理边界。
3. 能通过模板、配置和开关解决的问题，不进入定制开发流程。
4. 定制开发形成的通用能力，应优先回收为标准平台能力。

## 5. 用户角色

### 5.1 消费者

- 注册登录
- 浏览当前商家商品
- 下单支付
- 申请售后
- 评价商品

### 5.2 商家主账号与子账号

- 提交入驻资料
- 创建和管理子账号
- 维护店铺信息
- 维护小程序配置
- 发布和管理商品
- 处理订单和售后
- 查看经营数据和账单
- 按店铺、小程序、模块分配子账号权限

### 5.3 平台交付管理员

- 管理商户接入与开通状态
- 管理分类模板和页面模板
- 管理模板能力和功能开关
- 管理版本发布与回滚
- 查看平台与商户数据

### 5.4 平台管理员

- 账号权限管理
- 系统配置
- 版本管理
- 审计日志查看

## 6. 核心业务模式

### 6.1 商家与小程序关系

1. 一个商家主体可对应一个或多个店铺，一个店铺可绑定一个小程序 AppID。
2. 一个商家主体下可存在一个主账号和多个子账号。
3. 主账号负责商家整体配置、子账号管理和权限分配。
4. 子账号可按店铺、小程序、功能模块获得授权，维护不同的小程序。
5. 平台维护统一小程序代码模板，商家小程序基于模板生成和发布。
6. 商家可按店铺/小程序分别配置品牌、首页、客服、营销、支付展示和售后规则。
7. 商家数据独立存储或逻辑隔离，所有查询必须带商家边界；同一商家内部再按账号权限限制可见范围。

#### 6.1.1 账号与小程序映射规则

1. 商家主账号可查看本商家全部店铺、小程序、商品、订单和账单数据。
2. 子账号可被授权管理一个或多个指定店铺/小程序。
3. 子账号权限至少支持查看、编辑、发布、审核、导出五类动作控制。
4. 子账号如未被授权到某个店铺/小程序，不得查看该范围内的经营数据和配置数据。
5. 平台需支持按“商家主体 -> 店铺 -> 小程序 -> 子账号”四层关系进行权限校验。

#### 6.1.2 子账号角色模型

1. 商家主账号是商家主体的最高权限账号，默认拥有全部店铺、小程序和功能权限。
2. 子账号采用“账号信息 + 权限范围”模式，由商家自行维护账号资料和授权关系。
3. 权限至少分为两层：功能权限和数据范围权限。
4. 功能权限用于定义能否查看、编辑、发布、导出、管理；数据范围用于限定可操作的店铺、小程序和数据范围。
5. 高风险操作默认仅主账号可执行，若开放给子账号，必须额外审批或单独授权。

#### 6.1.3 授权范围规则

1. 授权范围至少支持商家级、店铺级、小程序级三种粒度。
2. 当子账号被授权到店铺级时，默认可访问该店铺绑定的小程序。
3. 当子账号被授权到小程序级时，仅可访问该小程序及其对应店铺的关联经营数据。
4. 导出、批量操作、发布等高影响动作应支持单独开关，不默认继承。
5. 子账号停用后应立即失效，已创建操作记录和审计日志必须保留。

### 6.2 商品分类模式

1. 平台提供标准商品分类模板，支持按行业预置多个版本。
2. 商家开通时可选择一个分类模板作为初始化分类。
3. 平台可在分类模板下提供对应的分类属性模板，用于初始化商品属性结构。
4. 商家可基于模板维护自己的商品分类树，包括新增、重命名、停用和排序。
5. 商家可基于平台属性模板复制并形成自己的分类属性，支持新增、删除、重命名、排序、设为必填或可选。
6. 商家分类调整需受平台规则约束，例如最大层级、禁用词、资质要求和敏感类限制。

#### 6.2.1 分类模板定义

1. 分类模板是平台提供的标准分类骨架，用于帮助商家快速初始化商品分类。
2. 一个分类模板至少包含模板名称、适用行业、模板版本、分类层级结构、默认排序、分类说明、适用品类约束和分类属性模板。
3. 平台可预置多个模板，例如通用零售模板、食品模板、美妆模板、家居模板。
4. 分类模板是初始化来源，不是商家线上分类的唯一来源；商家复制后形成自己的分类树。

#### 6.2.1.1 分类属性模板定义

1. 分类属性模板是挂载在分类模板或分类节点下的标准属性集合，用于帮助商家快速初始化商品参数。
2. 一个分类属性模板至少包含属性名称、属性编码、属性类型、是否必填、是否支持多选、可选值范围、排序规则和展示说明。
3. 平台提供的属性模板属于初始化参考，商家复制后形成自己的分类属性配置。
4. 商家自定义属性时，平台仍可保留基础属性规范边界，例如敏感词、特殊品类受控属性和必备资质属性。

#### 6.2.2 商家分类树规则

1. 商家分类树归属单一商家，不与其他商家共享。
2. 商家分类至少支持三级结构，最大层级由平台统一配置。
3. 商家可对分类进行新增、编辑、排序、启用、停用、合并和迁移。
4. 商家删除分类前，系统必须校验该分类下是否仍存在已上架商品。
5. 商家停用分类后，该分类不再对消费者展示，但历史商品和订单需保留分类快照。

#### 6.2.2.1 商家分类属性规则

1. 商家分类属性归属单一商家，不与其他商家共享。
2. 商家可基于平台属性模板初始化分类属性，也可在平台规则允许范围内新增自定义属性。
3. 商家分类属性至少支持输入型、单选型、多选型、数值型和文本说明型。
4. 商家可设置属性是否必填、是否前台展示、是否参与商品筛选、是否支持多值。
5. 商家删除或停用属性前，系统应校验是否已有商品使用该属性；若已有商品使用，应保留历史快照或要求先完成迁移。

#### 6.2.3 模板复制与演进规则

1. 商家首次开通时可选择一个平台模板进行初始化，也可在平台允许的情况下选择空白分类树。
2. 模板复制后，商家分类和分类属性与平台模板解耦，平台后续修改模板不会自动覆盖商家现有配置。
3. 若平台发布新版模板，可提供“对比差异后手动同步”能力，不允许强制覆盖商家分类或分类属性。
4. 模板同步仅允许新增推荐分类、推荐属性或更新说明，不应直接改动商家已绑定商品的分类结构和属性结构。

#### 6.2.4 平台管控边界

1. 平台可控制最大层级、命名规范、敏感词、禁售分类、资质要求、是否允许前台展示和基础属性规范边界。
2. 平台不可直接修改商家已生效的分类名称、排序、分类属性和前台展示结构，除非触发违规治理。
3. 对于受监管商品，平台可要求商家必须挂载到指定受控分类或补齐资质后方可发布。
4. 平台应支持查看商家分类树与模板的偏离情况，以及分类属性与属性模板的偏离情况，用于运营分析和治理。

### 6.3 商家定制化分层

| 层级 | 定义 | 示例 | 实现方式 |
| --- | --- | --- | --- |
| 标准配置 | 不改代码，仅配置参数 | Logo、主题色、首页楼层、导航、客服、分享卡片 | 后台配置 |
| 功能开关 | 在现有模板能力内启停功能 | 优惠券、积分、秒杀、预约、自提 | 开关控制 |
| 审批制扩展 | 基于预留扩展点增加局部能力 | 自定义首页模块、专属营销组件、第三方接口适配 | 评审后开发接入 |

### 6.4 定制化原则

1. 定制需求优先落在标准配置层。
2. 通用性较强的定制需求应沉淀为平台标准能力。
3. 审批制扩展不得破坏主模板结构和其他商家的稳定性。
4. 未经平台评审通过，不允许为商家直接改核心代码逻辑。

### 6.5 源码交付原则

1. 支持向平台方、甲方或实施方交付平台级完整源码，不支持按商家长期拆分源码交付。
2. 源码交付对象应具备自主部署、自主运维和二次开发能力。
3. 源码交付后，商家差异化仍应优先通过配置、功能开关和扩展点实现。
4. 第三方闭源依赖、商业授权组件和敏感密钥不纳入明文源码交付范围。

## 7. 业务流程概览

### 7.1 商家入驻与开通流程

1. 商家注册账号。
2. 提交主体与经营资质。
3. 平台处理接入资料。
4. 资料处理通过后签署协议并确认基础交易规则。
5. 商家绑定小程序 AppID、AppSecret 等接入信息。
6. 商家配置店铺基础信息和首版装修内容。
7. 平台完成配置校验和首版发布。
8. 商家上架商品并开始经营。

说明：对标准商家场景，若主体资料处理通过、AppID 接入验证通过、支付参数完整、基础配置已填写，则系统应支持“一键开通并发布首版小程序”。

### 7.2 商家定制化申请流程

1. 商家提交定制化需求。
2. 平台判定需求归属标准配置、功能开关或审批制扩展。
3. 若为标准配置，直接配置并预览。
4. 若为功能开关，确认适用范围后启用。
5. 若为审批制扩展，进入评审、排期、开发、测试、灰度、发布流程。
6. 发布后记录版本与变更范围。

### 7.3 模板版本发布流程

1. 平台研发产出模板版本。
2. 平台测试完成公共回归验证。
3. 运营或管理员选择目标商家范围。
4. 系统校验目标商家配置兼容性。
5. 支持灰度发布、全量发布和紧急回滚。
6. 发布结果记录到版本日志和商家操作日志。

### 7.4 用户交易流程

1. 用户进入某个商家的小程序。
2. 浏览当前商家的首页、分类和商品详情。
3. 加入购物车并提交订单。
4. 用户支付。
5. 商家发货。
6. 用户确认收货。
7. 订单完成。
8. 系统生成订单账单与退款对账记录。

### 7.5 售后流程

1. 用户提交退款或退货退款申请。
2. 商家审核。
3. 商家完成售后处理。
4. 退款完成后更新订单和账单状态。

### 7.6 商品发布流程

1. 商家创建或编辑商品草稿。
2. 商家补全商品基础信息、分类、库存、价格、图片和详情内容。
3. 商家在商品列表中选择单个或多个商品执行一键发布。
4. 系统先执行发布前校验，检查必填信息、分类有效性、库存价格规则、资质约束和图片完整性。
5. 校验通过的商品直接上架。
6. 校验失败的商品保持原状态。
7. 系统返回发布结果，区分成功和失败，并展示失败原因。

## 8. 功能需求

### 8.1 消费者小程序

#### 8.1.1 用户体系

- 支持手机号、微信授权登录。
- 支持收货地址管理。
- 支持账号注销。
- 支持用户协议和隐私政策展示确认。

#### 8.1.2 首页与店铺页

- 首页展示当前商家的轮播、导航、活动、推荐商品、公告等内容。
- 支持商家装修楼层配置，包含图片、商品组、公告、优惠券、分类入口等标准模块。
- 支持根据商家配置展示主题色、品牌文案、客服入口、营业状态。
- 当商家休息中或停业时，页面需展示明确状态和下单限制提示。
- 首页首屏应优先突出“当前位置/服务范围 + 履约承诺 + 搜索入口”，适配即时零售和生鲜到家场景。
- 首页信息层级建议采用“顶部服务信息 -> 搜索 -> 活动主视觉 -> 金刚区 -> 优惠与会场 -> 分类商品楼层 -> 推荐商品流”的结构。
- 商品楼层应以强图片卡片、价格直出、规格简述和快速加购为主，减少过深跳转。
- 对于生鲜、商超类商家，首页应支持时令会场、热销榜单、快手菜/套餐、品牌直采等运营区块。

#### 8.1.2.1 生鲜即时零售风格首页建议

- 参考盒马鲜生、七鲜这类即时零售首页的结构特征，平台应提供一套“生鲜即时零售版”首页模板，适合作为商家默认首页基线。
- 顶部区域建议包含收货地址、当前门店、最快送达时间、营业状态和搜索框，强化“附近门店 + 快速履约”感知。
- 第一屏建议至少包含一个主活动 Banner、一个优惠券或新人权益入口，以及 8 到 10 个高频频道入口，例如水果、蔬菜、肉蛋、水产、乳品、熟食、酒水、日用。
- 中部楼层建议支持“今日爆款”“时令推荐”“排行榜”“快手菜”“凑单专区”等分组，便于商家围绕高频采购场景运营。
- 推荐商品流建议采用双列或大图卡片流，突出商品图、活动价、原价、规格和销量/好评标签。
- 首页模板应保留跨行业适配能力；非生鲜商家可复用相同结构，但替换频道入口和活动楼层内容。

#### 8.1.2.2 生鲜类首页功能支持清单

- 支持地址选择、门店切换、配送范围识别和当前服务状态展示。
- 支持展示履约承诺信息，例如“最快送达时间”“起送门槛”“配送费说明”“营业中/休息中”。
- 支持首页搜索直达商品、分类、活动会场和品牌专区。
- 支持金刚区频道快捷入口，至少支持图标、名称、排序、角标、跳转目标配置。
- 支持新人券、满减券、限时券等优惠入口展示，并支持按用户状态决定是否展示。
- 支持主会场 Banner、次级活动位、横滑活动卡片和专题会场入口。
- 支持“时令推荐”“今日爆款”“热销榜”“快手菜/套餐”“凑单专区”“品牌直采”等标准商品楼层。
- 支持商品卡片直接展示价格、原价、规格、销量、库存状态、促销标签和快速加购按钮。
- 支持首页悬浮购物车、客服入口和返回顶部等悬浮操作入口。
- 支持缺货、售罄、休息中、超出配送范围等状态在首页商品卡和服务条中清晰提示。
- 支持首页按门店、配送区域、营业状态加载不同商品和活动数据，避免商家多个门店共用一套静态首页内容。
- 支持埋点统计首页曝光、点击、加购、券领取、搜索使用和楼层转化数据。

#### 8.1.2.3 非生鲜类首页建议

- 非生鲜类首页应更强调品牌表达、活动导购、分类浏览和内容化种草，不强依赖“附近门店 + 即时履约”作为首屏核心信息。
- 首页信息层级建议采用“品牌头图/品牌信息 -> 搜索 -> 核心分类入口 -> 活动会场 -> 新品/爆款 -> 品牌或内容专区 -> 推荐商品流”的结构。
- 对服饰、美妆、母婴、家居、数码、图书等非生鲜场景，首页可优先突出品牌调性、系列专题、搭配推荐、达人内容、会员权益和大促会场。
- 商品卡片可弱化即时送达文案，强化品牌、卖点、评价、销量、券后价或会员价等信息展示。
- 非生鲜首页应支持更大的视觉 Banner、更完整的专题会场和更强的内容化楼层，例如图文混排、选购指南、品牌故事、场景推荐。

#### 8.1.2.4 非生鲜类首页功能支持清单

- 支持品牌主视觉、品牌故事、品牌专区和系列专题入口配置。
- 支持首页搜索直达商品、分类、品牌、专题会场和内容专题。
- 支持分类频道入口、主推品类专区、新品专区、爆款专区和大促专区。
- 支持优惠券、满减、会员权益、积分专区等营销入口展示。
- 支持图文混排、品牌馆、搭配购、种草内容、选购指南等内容型楼层。
- 支持商品卡展示券后价、会员价、评价数、销量、核心卖点和收藏/加购入口。
- 支持按活动周期自动切换首页会场和楼层内容，满足大促、节庆、上新等运营需求。
- 支持首页按商家店铺、用户身份、会员等级和活动状态展示差异化内容。
- 支持首页悬浮客服、领券入口、会员入口和返回顶部等常用操作入口。
- 支持埋点统计首页曝光、点击、收藏、加购、领券、会场跳转和专题转化数据。

#### 8.1.2.5 生鲜类与非生鲜类首页模块对照

| 模块 | 生鲜类首页 | 非生鲜类首页 | 说明 |
| --- | --- | --- | --- |
| 顶部服务条 | 必配 | 可选 | 生鲜类强调地址、门店、配送范围、营业状态；非生鲜通常可弱化 |
| 品牌主视觉 | 可选 | 必配 | 非生鲜首页通常将品牌调性和系列主题放在首屏 |
| 搜索栏 | 必配 | 必配 | 两类首页都需要，默认都应支持热搜词和占位文案 |
| 金刚区频道 | 必配 | 必配 | 生鲜偏高频品类入口，非生鲜偏核心类目或系列入口 |
| 主 Banner/主会场 | 必配 | 必配 | 用于活动、专题或新品主推 |

#### 8.1.3 搜索与分类

- 支持商家内关键词搜索。
- 支持按当前商家的商品分类、价格、销量、上新时间筛选。
- 搜索结果仅返回当前商家的商品。
- 分类页展示以当前商家的分类树为准，支持一级、二级、三级分类切换。
- 当商家停用某分类时，前台分类入口自动隐藏，但已下单历史不受影响。

#### 8.1.4 商品详情

- 展示商品标题、图片、视频、价格、库存、规格、运费、销量、评价。
- 展示所属店铺信息、配送说明、售后规则、发货时效。
- 支持商家配置详情页服务承诺文案。
- 不展示其他商家商品入口。

#### 8.1.5 购物车

- 仅支持加入当前商家商品。
- 支持失效商品提示、库存校验、优惠试算。
- 商家切换后购物车按商家隔离。

#### 8.1.6 订单与支付

- 支持提交订单、优惠抵扣、运费计算、在线支付。
- 支持订单状态流转：待支付、待发货、待收货、已完成、已取消、售后中。
- 支持支付结果回跳和支付失败重试。

#### 8.1.7 售后与评价

- 支持仅退款、退货退款。
- 支持上传售后凭证、查看处理进度。
- 支持商品评价、追评、晒单。

#### 8.1.8 个人中心

- 查看订单、售后、优惠券、地址、客服入口。
- 展示当前商家会员权益信息，若商家未开通则不展示对应入口。

### 8.2 商家入驻

#### 8.2.1 商家注册

- 支持企业商家、个体商户两类主体。
- 支持联系人、手机号、登录密码设置。
- 支持短信验证码校验。

#### 8.2.2 入驻申请

- 支持填写主体信息、法人信息、经营分类、品牌信息。
- 支持上传营业执照、法人身份证、商标证书、授权文件、开户信息。
- 支持保存草稿。
- 支持提交后查看处理进度。

#### 8.2.3 接入处理管理

- 支持通过、驳回、补充资料。
- 驳回时必须填写原因。
- 处理过程需保留操作日志。
- 处理结果通过站内信、短信或邮件通知商家。

#### 8.2.4 协议与规则

- 支持在线查看并确认平台协议。
- 支持按分类模板或商家分类配置保证金和资质要求。
- 支持记录保证金缴纳状态。

#### 8.2.5 小程序接入

- 支持同一商家录入多个小程序 AppID、AppSecret、支付商户号、回调地址等信息。
- 支持校验接入信息完整性。
- 支持平台查看接入状态：未配置、待验证、已验证、已开通。
- 支持记录首次开通时间和当前绑定模板版本。
- 支持商家执行一键接入检测，自动校验 AppID、密钥、服务器域名、支付配置、隐私协议配置是否齐全。
- 对于检测通过的商家，支持一键开通能力；检测失败时需返回明确阻塞项和修复指引。
- 支持将不同 AppID 绑定到不同店铺，并配置对应维护负责人或子账号。

#### 8.2.6 一键开通与一键发布

- 支持商家在满足前置条件后，通过一个统一入口完成“开通店铺、初始化配置、绑定模板、提交首版发布”。
- 一键流程应自动完成标准配置初始化，包括主题默认值、首页默认模板、默认导航、默认客服配置。
- 一键流程应自动校验必填项，包括主体资料处理状态、协议确认状态、支付配置、基础装修配置、商品数量下限。
- 若校验通过，系统自动生成首版发布任务并进入发布流程。
- 若校验不通过，系统需按模块返回问题清单，例如“支付参数缺失”“首页未配置”“商品未上架”。
- 一键发布仅适用于标准模板和标准扩展范围；存在审批制扩展或高风险开关未完成评审时，不允许直接发布。

#### 8.2.7 一键开通页字段清单

| 分组 | 字段 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| 主体信息 | 商家名称 | 是 | 资料处理通过后默认带出，只读 |
| 主体信息 | 主体类型 | 是 | 企业/个体，资料处理通过后只读 |
| 店铺信息 | 店铺名称 | 是 | 默认取商家名称，可修改一次或按规则限制 |

| 店铺信息 | 店铺简介 | 否 | 展示在店铺页与分享文案 |

| 目标范围 | 目标店铺/小程序 | 是 | 一键开通和发布的目标对象 |
| 小程序接入 | AppID | 是 | 校验是否已绑定当前商家 |
| 小程序接入 | AppSecret | 是 | 仅录入或加密存储，不明文回显 |
| 小程序接入 | 支付商户号 | 是 | 用于支付能力开通 |
| 小程序接入 | 回调地址 | 是 | 支付与业务回调地址 |

- 页面应分为“基础信息”“接入配置”“模板配置”“发布校验”四个区域。
- 已处理通过且已有资料的字段应尽可能自动带出，减少重复录入。
- 密钥类字段必须脱敏展示，并支持重新上传或重新校验。

#### 8.2.8 一键开通流程细化

1. 商家进入一键开通页。
2. 系统自动拉取商家处理状态、协议状态、现有接入配置、默认模板配置。
3. 商家补全缺失字段并保存。
4. 商家点击“开始检测”后，系统执行接入和配置校验。
5. 若校验失败，系统按模块返回失败项并支持跳转修复。
6. 若校验通过，商家点击“一键开通并发布”。
7. 系统自动创建店铺、初始化默认配置、绑定模板版本、生成发布任务。
8. 发布完成后展示结果页，包括店铺状态、模板版本、发布时间、访问二维码。

#### 8.2.9 一键开通校验项清单

| 校验分类 | 校验项 | 失败示例 |
| --- | --- | --- |
| 主体校验 | 商家主体资料处理必须已通过 | 当前商家仍处于待处理状态 |
| 协议校验 | 平台协议、隐私协议必须确认 | 未勾选隐私协议 |
| 接入校验 | AppID、AppSecret、商户号、回调地址完整 | 支付商户号缺失 |
| 环境校验 | 小程序服务器域名、上传域名、合法域名配置完整 | 服务器域名未配置 |
| 模板校验 | 模板版本可用且与商家配置兼容 | 当前模板版本不支持该扩展组件 |
| 页面校验 | 首页模板、主题色、导航配置完整 | 首页模板未选择 |
| 商品校验 | 至少存在约定数量的已上架商品 | 当前无已上架商品 |
| 开关校验 | 高风险功能开关已完成评审 | 发票开关未完成评审 |
| 扩展校验 | 审批制扩展全部完成发布评审 | 自定义组件仍在待评审 |

- 首期建议将商品下限设置为“至少 1 个已完成发布校验且已上架商品”。
- 校验结果需区分阻塞项和提示项。
- 阻塞项必须修复后才能继续一键发布；提示项允许发布但需商家确认。

#### 8.2.10 一键发布结果与失败处理

- 发布成功后展示店铺状态、当前模板版本、发布流水号、发布时间、预览二维码、线上访问入口。
- 发布中断时需展示失败阶段，例如“接入校验失败”“模板打包失败”“发布提交失败”“平台回调失败”。
- 系统需提供“重新检测”“重新发布”“返回修改配置”三个操作入口。
- 对于重复失败的商家，系统应建议转人工支持或平台交付支持介入。
- 一键发布结果页需保留最近 10 次发布记录入口。

### 8.3 商家后台

#### 8.3.1 店铺管理

- 设置店铺名称、Logo、简介、客服信息、退货地址。
- 配置营业状态：营业中、休息中、停业整顿。
- 配置服务承诺、发货时效、售后说明。
- 支持同一商家维护多个店铺，并为每个店铺绑定独立小程序。

#### 8.3.2 小程序装修与配置中心

- 支持配置主题色、导航图标、首页轮播、首页楼层、底部导航文案。
- 支持配置分享标题、分享图、客服方式、公告内容。
- 支持预览当前配置效果。
- 支持保存草稿、提交发布、回滚至上一版本配置。
- 支持按店铺/小程序维度分别维护装修配置。
- 首页模板至少支持“通用商城版”和“生鲜即时零售版”两类风格。
- 首页模板至少支持“通用商城版”“生鲜即时零售版”“品牌零售版”三类风格。
- 生鲜即时零售版首页应支持配置顶部服务条、搜索框占位文案、履约承诺文案、金刚区频道、活动会场、热销榜单和推荐商品流。
- 品牌零售版首页应支持配置品牌头图、品牌文案、专题会场、图文内容楼层、会员权益入口和品牌专区。
- 首页标准模块至少包括：顶部服务信息、搜索栏、主 Banner、频道金刚区、优惠券卡片、公告条、活动横滑卡、商品分组楼层、排行榜楼层、品牌专区、推荐商品流、悬浮购物车/客服入口。
- 商家可按模块配置显示顺序、是否启用、跳转目标、商品来源和展示样式。
- 生鲜即时零售版首页还应支持配置门店展示信息、配送范围提示、送达时效文案、起送门槛文案、缺货替代提示和商品卡加购样式。
- 品牌零售版首页还应支持配置品牌主视觉、品牌故事、专题会场、内容型楼层、会员权益卡和收藏/分享样式。
- 对首页商品楼层，系统应支持按分类、商品组、活动、排行榜、手动选品四种来源装配数据。
- 对首页活动位，系统应支持按时间窗口自动上下线，避免商家频繁手工切换。
- 首页预览应支持按不同营业状态、门店和用户身份模拟查看，以便商家验证展示差异。

#### 8.3.3 账号与权限中心

- 支持商家主账号创建、启用、停用、重置子账号。
- 支持商家自行维护子账号名称、手机号、备注、负责业务等基础信息。
- 支持按店铺、小程序、菜单、功能动作分配权限。
- 支持设置子账号可查看的数据范围，例如仅查看指定小程序订单或指定店铺商品。
- 所有子账号的关键操作需记录审计日志。

#### 8.3.4 子账号角色与权限矩阵

| 权限维度 | 可配置项 | 说明 |
| --- | --- | --- |
| 账号类型 | 主账号、子账号 | 主账号默认全量权限，子账号按授权生效 |
| 功能权限 | 查看、编辑、发布、导出、管理 | 按模块独立配置 |
| 模块权限 | 店铺、小程序装修、商品、订单、售后、账务、数据看板 | 允许按模块勾选 |
| 数据范围 | 商家级、店铺级、小程序级 | 决定数据可见边界 |
| 高风险权限 | 发布、批量导出、账号管理、高风险开关 | 建议单独授权 |

- 平台不强制预置复杂角色矩阵，默认只提供主账号和子账号两类账号形态。
- 商家主账号可根据自身组织方式维护子账号信息和权限组合。
- 若需要行业通用模板角色，可作为可选预设能力提供，但不应作为系统唯一模式。

#### 8.3.5 子账号管理流程

1. 主账号创建子账号并填写姓名、手机号、登录账号、角色。
2. 主账号选择授权范围，可按店铺或小程序勾选。
3. 主账号勾选该子账号可操作的模块和动作权限。
4. 主账号确认后，系统向子账号发送开通通知。
5. 子账号首次登录后需完成密码设置或安全验证。
6. 主账号可随时调整授权范围、停用账号或重置登录信息。

#### 8.3.6 功能开关中心

- 支持启停优惠券、积分、秒杀、拼团、预约、自提等能力。
- 平台可按商家授权可用功能范围。
- 商家仅可启用已获授权的功能。

#### 8.3.7 功能开关范围细化

| 一级模块 | 二级能力 | 是否建议开关化 | 说明 |
| --- | --- | --- | --- |
| 用户中心 | 微信授权登录、手机号绑定、会员权益展示 | 是 | 不同商家开通节奏不同，适合按商家启停 |
| 首页装修 | 公告、优惠券入口、活动楼层、客服悬浮窗、弹窗 | 是 | 属于展示型能力，适合配置加开关 |
| 商品中心 | 视频展示、商品参数、服务承诺、预售标识 | 是 | 页面展示差异大，但逻辑相对稳定 |
| 营销中心 | 优惠券、满减、秒杀、拼团、积分、会员价 | 是 | 商家差异最明显，应作为标准开关域 |
| 交易中心 | 预约下单、自提、备注、发票申请、限购 | 是 | 对下单链路有影响，需支持联动校验 |
| 履约中心 | 物流配送、自提核销、同城配送、发货时效展示 | 是 | 与商家履约方式相关，适合开关治理 |
| 售后中心 | 仅退款、退货退款、换货入口、售后时效 | 是 | 与商家售后政策直接相关 |
| 支付中心 | 微信支付、余额抵扣、积分抵扣、组合支付 | 部分建议 | 涉及风控和账务校验，需平台强控 |
| 内容中心 | 店铺故事、帮助中心、协议展示、品牌页 | 是 | 以展示型内容为主 |
| 数据与会员 | 浏览埋点、会员积分、等级体系、成长值 | 部分建议 | 可开关，但需考虑数据一致性 |

- 功能开关需支持按商家、按模板版本、按环境、按生效时间设置。
- 功能开关需支持互斥关系和依赖关系校验，例如未开通优惠券时不可展示领券入口。
- 高风险功能开关需支持平台强控，商家不可自行关闭或开启，例如支付、账务校验、风控相关能力。
- 功能开关变更后需记录变更人、变更前后值、生效时间和影响范围。

#### 8.3.8 功能开关管理模型

| 维度 | 说明 | 示例 |
| --- | --- | --- |
| 开关编码 | 平台统一唯一标识 | `marketing.coupon.enable` |
| 开关层级 | 平台级、商家级、环境级、活动期临时级 | 平台默认关闭，商家A开启 |
| 开关类型 | 布尔开关、枚举开关、阈值开关、白名单开关 | 配送方式=`express/self_pickup` |
| 生效方式 | 立即生效、定时生效、灰度生效 | 2026-03-25 10:00 生效 |
| 默认值 | 模板默认值或平台默认值 | 默认关闭 |
| 风险等级 | 低、中、高 | 支付相关为高风险 |
| 控制权限 | 平台控制、商家可配、只读展示 | 账务相关仅平台控制 |

- 布尔开关适用于单纯启停类能力，例如积分、拼团、品牌页。
- 枚举开关适用于多模式互斥场景，例如配送方式、首页风格、客服方式。
- 阈值开关适用于限制类能力，例如限购数量、最低起送金额、免邮门槛。
- 白名单开关适用于试点场景，例如新营销玩法先对部分商家启用。

#### 8.3.9 功能开关依赖与互斥规则

| 当前开关 | 依赖项 | 互斥项 | 说明 |
| --- | --- | --- | --- |
| 优惠券 | 营销中心可用、订单优惠计算开启 | 无 | 未开通优惠计算时不可单独开优惠券 |
| 积分抵扣 | 会员积分体系、支付抵扣能力 | 部分商家自定义余额抵扣 | 需参与实付金额计算 |
| 秒杀 | 营销中心、库存锁定能力 | 部分预约商品模式 | 对库存扣减策略有要求 |
| 拼团 | 营销中心、订单扩展字段 | 秒杀同商品同时间段强互斥可选 | 避免同商品营销冲突 |
| 自提 | 门店/自提点配置、核销能力 | 纯物流配送可选互斥 | 影响地址和配送选择 |
| 发票申请 | 商家开票规则配置 | 无 | 需增加下单附加字段 |
| 换货入口 | 售后扩展流程 | 仅退款流程不互斥 | 需平台支持换货状态机 |

- 系统在商家后台保存开关前必须进行依赖校验。
- 若存在互斥开关同时开启，系统需阻止提交并给出明确冲突提示。
- 开关变更影响交易链路时，应在发布前生成影响评估结果，例如影响页面、接口、订单字段。

#### 8.3.10 功能开关权限与操作规则

- 平台管理员可创建、下线、冻结开关定义。
- 平台交付管理员可给商家授权可见和可用范围，但不可修改高风险开关底层逻辑。
- 商家主账号或被授权子账号仅可操作已授权且风险等级为低或中风险的开关。
- 高风险开关的启停必须走审批流，至少包含申请人、审批人、生效时间和回退方案。
- 生产环境开关变更必须保留操作日志，并支持按商家和时间检索。

#### 8.3.11 功能开关发布与回退规则

- 开关变更应支持草稿、待生效、生效中、已生效、已回退、已失效状态。
- 定时生效的开关在生效前允许撤销或修改。
- 高风险开关在生产环境变更前需提示受影响商家数量和关键链路。
- 开关生效失败时，应自动回退到上一有效值并通知平台管理员。
- 开关中心应支持查看最近一次生效结果、失败原因和回退记录。

#### 8.3.12 定制化申请中心

- 商家可提交定制需求，填写业务目标、期望页面、功能说明、期望上线时间。
- 系统需记录需求类型、审批状态、负责人、预计版本。
- 支持查看处理进度和历史记录。

#### 8.3.13 商品分类管理

- 支持商家在开通时选择平台分类模板并一键初始化分类树。
- 支持基于平台分类模板同步初始化分类属性模板。
- 支持查看模板原始结构与当前商家分类结构差异。
- 支持新增同级分类、子级分类、编辑分类名称、排序、启停状态。
- 支持配置分类图标、分类描述、前台是否展示。
- 支持为分类维护属性列表，包括属性名称、属性类型、是否必填、是否前台展示、是否参与筛选、可选值配置。
- 支持商家基于平台属性模板复制后再自定义分类属性。
- 支持分类属性排序、启停、删除和预览。
- 支持分类商品数量统计，显示各分类下商品数和上架商品数。
- 支持分类迁移，将一个分类下商品批量迁移到另一个分类。
- 删除分类前需校验是否存在商品；若存在商品，应要求先迁移或下架。
- 删除分类属性前需校验是否存在商品已使用该属性；若存在商品，应提示先处理历史数据或保留快照。
- 支持分类预览，查看分类页在小程序前台的展示效果。

#### 8.3.14 商品管理

- 支持商品新增、编辑、上下架、删除。
- 支持 SPU/SKU 管理。
- 商品发布时必须选择当前商家的商品分类。
- 平台应提供标准分类模板，商家可复制模板后按自身经营需要新增、删减、调整分类结构。
- 商品编辑时应支持根据当前分类自动加载该分类下的属性配置，供商家填写商品参数。
- 支持商品属性值录入，属性项来源于当前商家分类下的属性配置。
- 支持商品图片、视频、详情描述维护。
- 支持库存、价格、运费模板设置。
- 支持商品列表中按条件筛选草稿、已上架、已下架、发布失败商品。
- 支持商家选择多个符合条件的商品执行一键发布。
- 一键发布前需自动校验商品名称、分类、分类必填属性、主图、详情、SKU、价格、库存、配送信息等必填项。
- 校验通过时，系统直接上架商品。
- 校验失败时，系统需按商品维度返回失败原因，例如“未配置主图”“库存为 0”“分类已停用”“缺少必填属性”。
- 支持展示最近一次一键发布结果，包括成功数和失败数。
- 支持对失败商品一键跳转编辑，修正后再次发起发布。

#### 8.3.14.1 一键发布商品规则

| 规则项 | 规则说明 |
| --- | --- |
| 发布范围 | 支持单商品发布和多商品批量发布 |
| 可发布状态 | 草稿、已下架、发布失败后修正完成的商品 |
| 不可发布状态 | 已上架、库存锁定中的商品 |
| 校验方式 | 发布前同步校验，失败商品不进入后续发布流程 |
| 发布结果 | 校验通过即直接上架 |
| 失败反馈 | 按商品返回失败原因和修复建议 |
| 操作权限 | 仅主账号或被授权子账号可执行商品发布 |

- 一键发布属于正式系统功能，应在商品管理页提供明确入口，不依赖 skill 或外部工具实现。
- 一键发布应支持“全量校验后统一提交”和“校验通过的先提交”两种执行策略，默认采用后者提高效率。
- 若同次批量发布中存在部分失败，成功商品应正常进入已上架状态，失败商品保持原状态。

#### 8.3.15 订单管理

- 查看订单列表和详情。
- 支持发货、打印面单、填写物流单号。
- 支持异常订单备注和导出。

#### 8.3.16 售后管理

- 处理退款申请。
- 审核退货申请。
- 上传拒绝理由或协商结果。
- 查询售后历史记录。

#### 8.3.17 账务中心

- 查看账单明细和退款冲减。
- 查看账期和账单状态。
- 查看退款冲减明细和对账差异说明。

#### 8.3.18 数据看板

- 展示销售额、订单量、访客数、转化率、退款率。
- 展示首页装修点击数据、活动参与数据、配置版本数据。
- 支持按日、周、月查看趋势。

### 8.4 平台交付后台

#### 8.4.1 商家管理

- 查看所有商家资料和接入状态。
- 按状态筛选：待处理、资料待补充、已通过、已冻结、已停用。
- 支持冻结、解冻、清退商家。

#### 8.4.2 商品分类与品牌管理

- 配置平台标准商品分类模板。
- 支持平台维护多个行业分类模板，供商家开通时直接选用。
- 支持商家基于平台模板复制生成自己的商品分类树，并在授权范围内调整分类名称、层级和启用状态。
- 支持平台在分类模板下维护属性模板，供商家初始化商品参数结构。
- 支持平台配置分类模板对应的保证金、资质要求和发布约束。
- 商家自定义分类时不得突破平台设置的最大层级、禁用词和基础资质约束。
- 管理品牌库与授权资料。

#### 8.4.3 分类模板治理细化

- 平台可创建、编辑、停用、复制分类模板，并维护模板版本。
- 每个模板需支持记录适用行业、推荐商家类型、分类层级说明、默认排序规则。
- 平台可为分类节点维护属性模板，包括基础属性、建议属性和特殊品类受控属性。
- 平台可设置模板为“推荐使用”“禁止新商家使用”“历史保留”三种状态。
- 平台应支持查看各模板被多少商家采用、各模板偏离率、常见商家自定义分类情况。
- 平台应支持查看各属性模板被多少商家采用、属性偏离率和常见自定义属性情况。
- 平台可下发模板升级建议，但不直接覆盖商家现有分类树。

#### 8.4.4 模板与组件管理

- 管理小程序公共模板版本。
- 管理标准装修模块和扩展组件。
- 管理功能开关、默认值和适用范围。

#### 8.4.5 商家定制化管理

- 评估商家定制需求。
- 标记需求归类：标准配置、功能开关、审批制扩展。
- 记录评审结论、排期、交付版本和影响范围。

#### 8.4.6 版本发布管理

- 选择模板版本和目标商家。
- 支持兼容性校验、灰度发布、全量发布、回滚。
- 支持按商家查看当前运行版本和发布历史。
- 支持标准商家的一键首版发布和后续一键重发。
- 一键发布前必须完成模板兼容性校验、开关依赖校验、扩展组件校验、基础配置完整性校验。
- 一键发布失败时需输出失败阶段、错误原因和建议处理动作。

#### 8.4.7 数据查看中心

- 查看商户维度的小程序接入状态、发布状态、版本状态和基础经营数据。
- 查看商品、订单、退款、访客、支付等汇总数据。
- 支持按商户、店铺、小程序、时间范围筛选查看数据。
- 支持查看商品一键发布批次结果，例如成功数、失败数、失败原因分布。

#### 8.4.8 内容与文档管理

- 管理公告、协议文档、帮助中心。
- 管理交付说明、接入规范和发布指引。

#### 8.4.9 权限与审计

- 支持 RBAC 权限模型。
- 支持角色、菜单、数据权限控制。
- 支持按商家主体、店铺、小程序、子账号范围进行权限校验。
- 所有关键操作需记录审计日志。

#### 8.4.10 数据权限矩阵细化

| 数据域 | 主账号规则 | 子账号规则 | 平台侧要求 |
| --- | --- | --- | --- |
| 店铺与小程序配置 | 默认可见全部 | 仅可见授权范围 | 修改需留痕 |
| 商品与分类数据 | 默认可见全部 | 仅可见授权店铺/小程序 | 批量操作需单独授权 |
| 订单与售后数据 | 默认可见全部 | 仅可见授权店铺/小程序 | 导出需记录审计日志 |
| 账单与对账数据 | 默认可见全部 | 仅可见被授权范围 | 金额字段支持脱敏 |
| 发布与操作记录 | 默认可见全部 | 仅可见授权范围 | 回滚权限默认仅主账号 |

- 平台应支持功能权限和数据范围分离配置，避免“有菜单就能看全部数据”。
- 子账号跨多个店铺/小程序时，查询结果应按授权范围自动过滤。
- 涉及金额、手机号、地址等敏感字段时，应支持字段级脱敏策略。

### 8.5 支付与账务

#### 8.5.1 支付能力

- 支持微信支付等主流小程序支付方式。
- 支持支付回调、对账、补单。
- 支持优惠抵扣后实付金额计算。

#### 8.5.2 订单规则

- 单个订单仅允许包含当前商家商品。
- 本期不支持跨商家购物车、合单和拆单。
- 订单状态按商家订单独立流转。

#### 8.5.3 账单与对账

- 平台按订单维度生成账单明细。
- 本期不收取平台佣金。
- 商家收入不计入平台收入口径，平台仅保留交易记录、账单记录和退款对账记录。
- 支持退款后自动冲减账单金额。

### 8.6 共享代码与定制化能力

#### 8.6.1 租户识别

- 小程序启动时根据 AppID 或商家标识识别当前商家租户。
- 所有接口请求必须自动带出商家标识。
- 未识别商家或商家状态异常时，禁止进入交易流程。

#### 8.6.2 配置加载规则

- 配置优先级为：平台默认配置 < 商家配置 < 活动期临时配置。
- 配置读取失败时需回退至最近一次有效配置。
- 配置变更需支持版本化和审计记录。

#### 8.6.3 扩展点机制

- 首页、商品详情、个人中心等页面需预留标准扩展区域。
- 审批制扩展仅允许在预留扩展点内注入能力。
- 扩展组件需声明适用商家、适用模板版本和依赖能力。

#### 8.6.4 扩展点范围细化

| 扩展层级 | 建议扩展点 | 典型场景 | 约束 |
| --- | --- | --- | --- |
| 页面级 | 首页楼层区、商品详情营销区、个人中心中部模块、店铺页活动区 | 商家专属活动入口、品牌专区、定制会员入口 | 不允许替换页面主框架和公共导航 |
| 组件级 | Banner 组件、商品分组组件、图文混排组件、公告组件、营销卡片组件 | 自定义首页模块、专属促销卡片 | 必须遵守统一组件协议和样式规范 |
| 交互级 | 按钮点击后跳转、自定义弹窗、分享逻辑、客服唤起 | 特定活动跳转、品牌弹窗、渠道分享页 | 不允许绕过平台鉴权和埋点 |
| 交易级 | 下单前校验、地址补充校验、配送方式附加校验 | 特殊商品预约、区域限制、企业采购附加字段 | 不允许改写核心价格、库存、支付计算链路 |
| 履约级 | 发货信息补充、核销字段扩展、物流回传适配 | 自提核销码、供应商单号映射 | 需保证主订单状态机不变 |
| 售后级 | 售后申请附加字段、凭证类型扩展、原因选项扩展 | 特殊品类售后问卷、安装类商品凭证上传 | 不允许绕过平台售后时效规则 |
| 接口级 | 第三方 CRM、ERP、客服、短信、BI 接口适配 | 商家已有系统对接 | 通过标准适配层接入，不直连业务核心服务 |
| 数据级 | 埋点事件扩展、报表字段扩展、标签扩展 | 商家自定义看板、专属经营分析 | 公共主数据结构不得被破坏 |

- 页面级扩展点应以 `slot` 或区域编码方式管理，至少包含首页顶部、首页中部、商品详情营销区、个人中心服务区。
- 组件级扩展点应定义统一入参、出参、样式约束、埋点规范和降级策略。
- 交易级扩展点必须采用前置校验或后置附加信息模式，不允许直接篡改核心订单计算逻辑。
- 接口级扩展点必须通过集成层或网关实现，禁止商家扩展代码直接访问核心数据库。

#### 8.6.5 扩展点生命周期管理

- 每个扩展点需有唯一编码、所属模块、支持版本、负责人和风险等级。
- 每个扩展组件需声明启用商家、依赖功能开关、回退方案和监控指标。
- 扩展组件上线前需经过兼容性测试、性能验证和安全审查。
- 平台需支持按扩展点查看启用商家列表、最近发布时间、失败记录和回滚记录。

#### 8.6.6 扩展点注册规范

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 扩展点编码 | 是 | 平台唯一编码，如 `page.home.middle.slot01` |
| 扩展层级 | 是 | 页面级、组件级、交互级、交易级、接口级等 |
| 所属模块 | 是 | 首页、商品、订单、售后、会员等 |
| 挂载位置 | 是 | 页面区域、接口钩子或校验节点 |
| 输入参数协议 | 是 | 统一的入参结构和字段说明 |
| 输出结果协议 | 否 | 返回结构、跳转结果、附加字段等 |
| 支持模板版本 | 是 | 最低支持版本、最高兼容版本 |
| 风险等级 | 是 | 低、中、高 |
| 降级方案 | 是 | 扩展失败时展示默认模块或忽略扩展 |
| 监控指标 | 否 | 渲染耗时、错误率、调用量等 |

- 所有扩展点必须先注册后使用，未注册扩展点不得上线。
- 扩展点编码命名建议采用 `层级.模块.区域.序号` 规则，便于治理和排查。
- 同一页面区域允许挂载多个扩展组件时，必须定义优先级和排序规则。
- 高风险扩展点需由平台管理员审批后才能开放给商家使用。

#### 8.6.7 扩展组件运行规则

- 扩展组件运行时不得阻塞主页面核心渲染超过约定阈值。
- 扩展组件异常时应按降级方案处理，不得导致首页、商品详情、下单页整体不可用。
- 扩展组件必须接入统一埋点、统一日志和统一异常上报体系。
- 扩展组件若依赖第三方接口，需配置超时、重试和熔断策略。
- 扩展组件若读取商家配置，必须通过统一配置服务获取，不得本地硬编码。

#### 8.6.8 扩展点审批与发布规则

- 页面展示类扩展可由平台交付管理员发起审批，交易类和接口类扩展必须由平台管理员或技术负责人审批。
- 审批单需包含适用商家、业务目标、风险说明、回退方案、验证结果。
- 扩展发布前需进行目标商家兼容性校验，包括模板版本、开关依赖、配置完整性。
- 扩展支持按商家灰度发布、按页面灰度验证、按时间窗口发布。
- 扩展回滚后应恢复至上一有效组件版本或默认平台组件。

#### 8.6.9 版本兼容性

- 模板版本升级前需校验商家配置、组件和功能开关是否兼容。
- 不兼容时禁止发布，并提示具体阻塞项。
- 支持按商家回滚到上一个稳定版本。

#### 8.6.10 变更治理

- 商家配置变更和模板升级应区分管理。
- 高风险变更需二次确认并记录审批人。
- 发布失败时需自动告警。

### 8.7 源码交付与私有化部署

#### 8.7.1 交付范围

- 小程序前端源码。
- 商家后台源码。
- 平台交付后台源码。
- 后端服务源码。
- 数据库初始化脚本和基础表结构说明。
- 配置文件模板、环境变量说明和部署手册。
- 接口文档、版本说明、操作手册。

#### 8.7.2 不交付范围

- 生产环境真实密钥、证书和账号。
- 未取得转授权资格的第三方商业组件源码。
- 第三方云服务本身。
- 单商家专属长期维护分支。

#### 8.7.3 私有化部署目标

1. 支持将整套小程序交付与发布平台部署在客户自有环境中独立运行。
2. 支持客户在私有环境内完成商户接入、模板配置、小程序发布和数据查看。
3. 支持客户使用自有网络、数据库、对象存储、日志系统和运维体系。
4. 支持源码交付后的二次开发，但整体架构仍保持统一模板和统一发布机制。

#### 8.7.3.1 小微企业低成本部署设计原则

1. 首期优先保障“能开通、能发布、能看数据”，不强制一次性建设高可用全套基础设施。
2. 优先复用客户现有云资源或托管服务，减少自建数据库、对象存储、日志集群和 CI/CD 成本。
3. 默认按“单客户、低并发、少量管理员”的现实场景设计，避免为未来大规模场景提前采购。
4. 生产环境与非生产环境必须隔离，但测试环境和预发环境可在小微场景下合并建设。
5. 非核心能力按需启用，例如短信、邮件、复杂监控、数据大屏等可后置接入。
6. 部署方案必须保留平滑升配路径，后续从轻量版升级到标准版时不应要求重写业务代码。
7. 若客户缺少发布自动化能力，平台可提供 CI/CD 服务作为可选交付项，而不是首期强制建设项。

#### 8.7.4 部署模式

| 模式 | 适用场景 | 特点 |
| --- | --- | --- |
| 轻量私有化部署 | 小微企业、预算敏感客户、单客户交付初期 | 以低成本上线为目标，建议 1 套生产 + 1 套测试/预发合并环境，优先复用托管基础设施 |
| 单实例部署 | 演示环境、小规模试点 | 部署简单，适合快速验证 |
| 标准多环境部署 | 正式交付项目 | 区分开发、测试、预发、生产环境 |
| 高可用部署 | 对稳定性要求较高的客户 | 服务多实例、负载均衡、主从或集群部署 |
| 专网/内网部署 | 政企或高安全场景 | 不依赖公网访问，需提前规划外部接口连通方式 |

- 小微企业或预算敏感客户默认推荐“轻量私有化部署”作为首期交付基线。
- 中大型项目默认推荐“标准多环境部署”作为项目交付基线。
- 若客户对稳定性、审计、安全有更高要求，推荐采用高可用部署。

#### 8.7.4.1 轻量私有化部署方案（小微企业低成本版）

| 组件 | 推荐方案 | 低成本说明 |
| --- | --- | --- |
| 应用服务 | 1 台应用服务器或 1 组容器主机 | 统一承载平台交付后台、商家后台、API 服务和定时任务，首期不强制拆分多节点 |
| 数据库 | 客户现有托管 MySQL/PostgreSQL | 优先复用已有数据库实例，减少自建数据库运维成本 |
| 缓存 | 单实例 Redis 或等价托管缓存 | 仅承担登录态、配置缓存和基础队列能力 |
| 对象存储 | 客户现有对象存储或兼容服务 | 用于商品图片、装修素材、二维码等文件，不建议自建文件服务器 |
| 日志 | 本地文件 + 轻量采集，或接入云日志服务 | 首期不强制建设 ELK 等重型日志平台 |
| 监控 | 主机监控 + 应用健康检查 + 基础告警 | 满足服务可用性巡检即可，复杂 APM 可后置 |
| 发布方式 | Docker Compose 或脚本化部署 | 交付简单，便于小团队维护，不强制引入 Kubernetes |

- 轻量版首期建议采用“单生产环境 + 单测试/预发合并环境”。
- 若客户二次开发较少，开发环境可由研发本地环境承担，不强制单独采购开发服务器。
- 小程序发布、商家接入、数据查看三条主链路必须可用，其余增强能力可按预算分期建设。

#### 8.7.4.2 轻量方案适用边界与升级路径

- 适用于单客户私有化、商家数量较少、日常发布频次低、运维团队较小的项目。
- 不适用于大促高并发、多团队并行发版、7x24 高可用、强审计留痕或专网合规要求高的项目。
- 当商家数量、数据量或发布频次明显上升时，应升级为标准多环境部署。
- 升级路径建议为：轻量私有化部署 -> 标准多环境部署 -> 高可用部署。
- 升级过程应以扩容基础设施和拆分部署单元为主，不应要求重构核心业务模块。

#### 8.7.5 环境规划

| 环境 | 主要用途 | 是否必须 |
| --- | --- | --- |
| 开发环境 | 二次开发、自测、联调 | 建议 |
| 测试环境 | 功能测试、回归测试、配置验证 | 必须 |
| 预发环境 | 接近生产的发布演练和验收环境 | 建议 |
| 生产环境 | 正式对外服务 | 必须 |

- 测试环境和生产环境必须隔离数据库、缓存、对象存储和日志配置。
- 生产环境不应直接使用测试环境的密钥、域名、回调地址和存储桶。
- 预发环境应尽量模拟生产网络和中间件版本，便于提前发现发布问题。
- 小微企业低成本方案可采用“测试兼预发”的合并环境，但生产环境必须独立。
- 若交付范围仅包含标准配置与少量模板调整，可不单独建设远程开发环境。

#### 8.7.6 私有化部署架构建议

| 层级 | 组成 | 说明 |
| --- | --- | --- |
| 接入层 | 网关、负载均衡、HTTPS 终端 | 负责请求入口和安全控制 |
| 应用层 | 平台交付后台、商家后台、后端业务服务 | 承载模板、发布、账号、数据查看等功能 |
| 数据层 | MySQL/PostgreSQL、Redis、对象存储 | 保存商户、配置、发布记录、素材等数据 |
| 集成层 | 短信、邮件、支付、日志、监控接口 | 替换为客户自有或指定服务 |
| 运维层 | CI/CD、监控告警、备份、审计 | 保障持续发布和运维可控 |

- 私有化部署后，前端和后端均应通过配置切换环境，不允许硬编码公网地址。
- 配置中心应支持分环境、分商户、分小程序加载配置。
- 平台可按项目需要提供 CI/CD 服务，用于构建、部署、回滚和发布流程标准化。

#### 8.7.6.1 低成本部署拓扑建议

1. 接入层、应用层可先部署在同一台应用主机上，通过 Nginx 或网关统一入口。
2. 数据库、对象存储、短信、日志等优先复用客户已有托管服务，避免重复采购。
3. 商家后台与平台交付后台优先共用统一后端服务，通过账号和权限模型隔离。
4. 定时任务、发布任务、消息通知在低成本阶段可与主应用同机部署，但需保留后续拆分能力。
5. 若客户后续商家数量增长，可优先拆分发布任务服务和后台服务，再考虑整体微服务化。

#### 8.7.7 外部依赖替换能力

| 依赖类型 | 私有化要求 |
| --- | --- |
| 数据库 | 支持替换为客户指定数据库实例 |
| 对象存储 | 支持替换为客户私有对象存储或兼容服务 |
| 缓存 | 支持替换为客户 Redis 或等价组件 |
| 短信/邮件 | 支持替换为客户采购的通知服务 |
| 支付回调 | 支持配置客户专属回调域名和证书 |
| 日志与监控 | 支持接入客户已有日志平台和监控平台 |

- 所有外部依赖应通过配置接入，不应依赖代码层面的硬切换。
- 若客户环境不提供某类外部能力，应允许关闭对应功能或提供替代方案。

#### 8.7.8 交付要求

- 源码应按模块划分清晰，支持独立构建和部署。
- 交付版本需有明确版本号、分支标识、发布日期和变更记录。
- 交付包需可在标准环境完成部署，不依赖研发本地私有环境。
- 需提供初始化部署文档、升级文档、回滚文档和常见故障处理说明。
- 若客户有需要，可提供配套 CI/CD 服务或 CI/CD 配置模板，支持构建、部署、回滚和环境切换。

#### 8.7.9 部署实施内容

- 环境准备清单：服务器、数据库、缓存、对象存储、域名、证书、网络策略。
- 安装部署清单：应用包、配置文件、数据库脚本、环境变量、启动脚本。
- 初始化清单：平台管理员账号、基础模板、分类模板、默认配置、示例商户。
- 联调清单：小程序合法域名、支付回调、短信/邮件通知、素材上传、日志采集。
- 验收清单：登录、模板管理、商户接入、一键开通、小程序发布、数据查看、权限控制。
- 对小微企业交付时，应额外提供轻量部署包、默认资源规格建议、升配建议和成本控制说明。
- 若包含 CI/CD 服务，应补充流水线配置清单、发布权限配置、回滚策略和环境变量管理清单。

#### 8.7.10 客户与交付方职责边界

| 事项 | 交付方负责 | 客户负责 |
| --- | --- | --- |
| 源码与部署包 | 提供 | 接收并管理 |
| 部署文档与实施指导 | 提供 | 按文档准备环境 |
| 环境资源 | 提供资源要求 | 提供服务器、网络、数据库等 |
| 外部依赖接入 | 提供接入规范 | 提供可用账号、密钥、域名 |
| CI/CD 服务（可选） | 提供流水线方案、配置或代建服务 | 提供代码仓库、执行环境和发布权限 |
| 上线验证 | 配合验证关键流程 | 组织业务验收 |
| 后续运维 | 可提供支持方案 | 日常运行和监控处理 |

#### 8.7.11 升级与版本管理

- 私有化版本需保留版本号、发布日期、变更说明和对应数据库变更记录。
- 升级前应先在测试或预发环境完成验证，再进入生产环境。
- 升级应支持备份、回滚和版本对照，避免直接覆盖旧版本。
- 客户若做二次开发，建议以分支方式管理，并定期评估与主干版本的合并策略。

#### 8.7.12 安全与运维要求

- 支持测试、预发、生产多环境部署。
- 支持配置中心、对象存储、短信、支付、日志等外部依赖替换。
- 支持租户隔离、数据备份、日志审计和权限控制。
- 私有化部署版本应保留统一模板和统一扩展机制。

#### 8.7.13 私有化部署验收标准

1. 客户可在自有环境完成系统安装、启动和基础配置。
2. 商户可在私有化环境完成接入、模板选择、一键开通和小程序发布。
3. 平台交付后台可正常查看商户接入状态、发布状态和基础数据。
4. 主账号/子账号权限、数据范围、审计日志在私有化环境中正常生效。
5. 升级、回滚、配置切换和日志查看能力可在约定环境中验证通过。
6. 若采用轻量私有化部署，客户可在“1 套生产 + 1 套测试/预发合并环境”下稳定运行核心链路。
7. 轻量版部署完成后，应可在不改动业务代码的前提下平滑升级为标准多环境部署。
8. 若交付范围包含 CI/CD 服务，流水线应可完成标准构建、部署、回滚和发布记录留痕。

#### 8.7.14 交付套餐说明

#### 8.7.14.1 套餐总览

| 套餐 | 适用客户 | 部署资源建议 | 交付重点 | 升级方向 |
| --- | --- | --- | --- | --- |
| 轻量版 | 小微企业、预算敏感客户、首期试运行项目 | 1 套生产 + 1 套测试/预发合并环境 | 先满足开通、发布、数据查看三条主链路 | 可升级到标准版 |
| 标准版 | 已明确长期运营、需要规范交付和稳定发版的客户 | 开发、测试、预发、生产分环境 | 完整交付、规范发版、标准运维，可选配 CI/CD 服务 | 可升级到高可用版 |
| 高可用版 | 对稳定性、安全、审计要求高的客户 | 多实例应用、独立中间件、主从或集群 | 高稳定、高审计、强灾备能力，可配套完整 CI/CD 服务 | 作为目标形态长期运行 |

#### 8.7.14.2 轻量版交付说明

| 维度 | 说明 |
| --- | --- |
| 适用客户 | 小微企业、单品牌客户、首期预算有限但希望先快速上线的小程序项目 |
| 部署资源建议 | 1 台应用服务器或 1 组轻量容器主机；1 个托管数据库；1 个 Redis；1 个对象存储；1 套生产环境 + 1 套测试/预发合并环境 |
| 交付内容 | 平台级源码包、部署脚本、初始化脚本、轻量部署手册、模板配置说明、商家开通与发布操作说明 |
| 客户需准备内容 | 服务器或云主机、数据库、缓存、对象存储、域名和证书、小程序 AppID、支付商户号、基础运维联系人 |
| 不包含内容 | 高可用集群、专网部署改造、复杂监控平台、专属灾备体系、7x24 运维值守、深度二次开发实施 |
| 适合的交付节奏 | 先完成系统部署和首个商家上线，再按实际运营情况逐步补充监控、日志、扩容和扩展功能 |
| 升级路径 | 当商家数量、发布频次或数据量增长时，升级为标准版，优先补齐独立测试、预发环境和发布流程治理 |
| CI/CD 服务 | 默认不强制配置，可按客户需要提供轻量流水线模板或代建服务 |

- 轻量版的核心目标是压低首期部署成本，而不是一次性把基础设施做满。
- 对外沟通时应明确：轻量版适合“先上线、再扩容”，不承诺高并发和复杂容灾能力。

#### 8.7.14.3 标准版交付说明

| 维度 | 说明 |
| --- | --- |
| 适用客户 | 计划长期运营、需要规范测试发布流程、需要多个商家持续接入的平台项目 |
| 部署资源建议 | 开发、测试、预发、生产四类环境；应用服务与发布任务可分离部署；数据库、缓存、对象存储采用独立实例或独立资源池 |
| 交付内容 | 平台级源码包、标准部署包、环境初始化脚本、升级回滚手册、版本发布说明、模板与分类模板初始化方案、运维交接清单 |
| 客户需准备内容 | 多环境服务器资源、数据库与缓存实例、对象存储、日志采集能力、发布账号、域名证书、测试与验收人员 |
| 不包含内容 | 专网隔离改造、双活容灾、跨地域灾备、强监管审计定制、非标准第三方系统深度集成 |
| 适合的交付节奏 | 按“环境准备 -> 系统部署 -> 联调验证 -> 首版上线 -> 运维交接”的标准项目节奏实施 |
| 升级路径 | 若后续对稳定性、审计和灾备要求提升，可升级为高可用版，重点增加多实例、集群化和更完整的运维体系 |
| CI/CD 服务 | 可作为推荐配套项提供，支持标准构建、环境部署、回滚和发布留痕 |

- 标准版应作为多数正式项目的默认报价基线。
- 若客户没有明确预算约束，但希望系统后续持续用下去，优先推荐标准版而不是轻量版。

#### 8.7.14.4 高可用版交付说明

| 维度 | 说明 |
| --- | --- |
| 适用客户 | 政企客户、中大型集团客户、强审计场景、业务连续性要求高的客户 |
| 部署资源建议 | 应用多实例部署；网关与负载均衡；数据库主从或集群；独立缓存集群；对象存储高可用方案；独立日志、监控和备份体系 |
| 交付内容 | 标准版全部内容 + 高可用架构方案、容量规划建议、故障切换说明、备份恢复方案、审计与安全策略建议 |
| 客户需准备内容 | 多可用区或多节点资源、网络与安全策略、日志监控平台、备份资源、运维与安全负责人、变更管理流程 |
| 不包含内容 | 客户内部安全制度建设、等保测评实施、第三方安全厂商服务、长期驻场运维服务，除非另行约定 |
| 适合的交付节奏 | 在标准部署验证通过后，再分阶段完成多实例扩容、容灾演练、审计接入和正式切流 |
| 升级路径 | 一般作为最终运行形态，可继续按监管要求补充专网部署、双活、异地灾备和更细的审计能力 |
| CI/CD 服务 | 建议纳入标准交付范围，支撑多环境发布、审批、回滚和审计留痕 |

- 高可用版适合有明确稳定性和合规要求的客户，不建议对预算敏感的小项目直接采用。
- 若客户当前需求不高，但采购流程要求先报高规格方案，可同步给出“标准版起步、后续升高可用”的分阶段实施建议。

#### 8.7.14.5 套餐选择建议

1. 若客户首先关心“预算低、尽快上线”，优先推荐轻量版。
2. 若客户首先关心“规范交付、后续持续使用”，优先推荐标准版。
3. 若客户首先关心“稳定性、安全、审计、容灾”，优先推荐高可用版。
4. 若客户当前预算有限但未来确定会扩容，应在轻量版合同中提前约定升配边界和升级方式。
5. 所有套餐都应明确“交付的是平台级系统能力”，不包含每个商家的长期独立分叉维护。

#### 8.7.14.6 对外沟通简版

#### 8.7.14.6.1 一句话说明

- 轻量版：适合预算敏感的小微企业，先低成本上线，先把开通、发布、数据查看跑起来。
- 标准版：适合准备长期运营的正式项目，交付更规范，环境更完整，后续更稳。
- 高可用版：适合对稳定性、安全和审计要求高的客户，按长期生产系统标准建设。

#### 8.7.14.6.2 客户沟通版总表

| 套餐 | 适合谁 | 客户最直接的价值 | 需要客户准备什么 | 不适合什么场景 |
| --- | --- | --- | --- | --- |
| 轻量版 | 小微企业、初创团队、先试运行的客户 | 花更少的钱先上线，后面再按业务增长逐步扩容 | 基础云资源、小程序账号、支付账号、域名证书 | 大促高并发、复杂审计、强灾备要求 |
| 标准版 | 计划长期运营、需要规范交付的客户 | 环境更完整，发布更稳，后续扩展更顺 | 多环境资源、测试验收人员、基础运维能力 | 极低预算、只想临时试跑的项目 |
| 高可用版 | 政企客户、中大型客户、强监管客户 | 稳定性更高，故障风险更低，审计和备份更完善 | 多节点资源、安全和运维体系、监控备份能力 | 预算有限、首期只求快速上线的项目 |

#### 8.7.14.6.3 推荐话术

- 如果客户说“我们预算有限，先上了再说”，推荐轻量版。
- 如果客户说“这个系统后面会长期用，不能太随意”，推荐标准版。
- 如果客户说“我们对安全、稳定、审计有明确要求”，推荐高可用版。
- 如果客户暂时预算不高，但未来确定会扩容，建议先签轻量版，同时在合同里约定后续升级到标准版的方式。

#### 8.7.14.6.4 沟通边界

1. 三个套餐交付的都是同一套平台能力，差别主要在部署规格、环境完整度和运维保障强度。
2. 轻量版不是功能缩水版，而是基础设施和交付规格更轻。
3. 标准版不是高可用版，若客户明确要求多实例、灾备、强审计，应单独升级方案。
4. 所有套餐都默认基于统一代码主干交付，不承诺为单个商家长期维护独立分叉版本。

## 11. 关键业务规则

1. 一个商家主体可绑定多个店铺和多个小程序 AppID。
2. 所有商家小程序必须基于统一代码模板构建，不允许形成长期商家专属分支。
3. 商家差异优先通过配置、功能开关、扩展组件实现。
4. 审批制扩展必须经过平台评审、测试和发布确认。
5. 商家之间数据必须严格隔离，接口、缓存、日志导出均不得串数据。
6. 同一商家主体下的子账号仅可访问被授权店铺/小程序范围内的数据和配置。
7. 单个订单仅允许购买当前商家商品，本期不支持跨商家交易。
8. 商家被冻结后，不可新增商品，但需继续处理存量订单售后。
9. 已出账订单发生退款时，需生成负向账单或下期冲减。
10. 商家配置变更不应影响其他商家的小程序页面和交易逻辑。
11. 公共模板升级不得覆盖商家已生效的有效配置。
12. 平台提供的分类属性模板仅作为初始化来源，商家复制后可在规则范围内自定义维护。
13. 源码交付以平台级代码主干为准，不支持每个商家独立维护长期分支。
14. 私有化部署后若产生个性化开发，仍应优先回归主干能力沉淀。
15. 一键开通和一键发布仅适用于满足标准前置条件的商家，不适用于存在未完成评审项或特殊定制扩展的商家。
16. 一键发布执行前必须通过接入校验、配置校验、商品校验和版本兼容性校验。
17. 商品一键发布仅对有发布权限的主账号或子账号开放，且必须经过商品完整性校验。
18. 商品一键发布失败不得影响同批次其他校验通过商品的正常发布或上架。

## 12. 非功能需求

### 12.1 性能要求

- 小程序首页接口平均响应时间小于 2 秒。
- 下单、支付、库存校验等关键交易接口平均响应时间小于 1 秒。
- 支持大促期间商家并发访问和订单峰值扩展。

### 12.2 安全要求

- 敏感信息加密存储。
- 关键接口需鉴权、签名和幂等校验。
- 支持防刷、防重、防越权。
- 商家级配置和交易数据必须带租户边界校验。

### 12.3 稳定性要求

- 核心交易服务可用性不低于 99.9%。
- 订单、支付、账单、配置发布链路需支持幂等处理。
- 配置中心故障时应能回退到最近稳定配置。

### 12.4 可运维性要求

- 支持日志查询、异常告警、链路追踪。
- 支持模板版本、商家版本、配置版本三类版本记录。
- 支持按商家查看当前运行版本、发布时间和回滚记录。
- 支持源码交付版本、部署版本和运行版本之间的映射关系查询。

## 16. 验收标准

1. 商家可以独立完成注册、入驻申请、资料补交、小程序接入和进度查询。
2. 平台交付管理员可以完成接入处理、商户管理、分类模板配置、模板管理、发布管理和数据查看。
3. 商家资料处理通过后可完成店铺开通、装修配置、功能启用并上架商品。
4. 用户可在任一商家的小程序内完成当前商家商品的浏览、下单、支付与售后。
5. 不同商家小程序使用同一套代码模板发布，但页面配置和业务数据彼此隔离。
6. 商家可通过标准配置、功能开关或审批制扩展满足定制化需求。
7. 系统可生成商家维度账单和退款对账记录。
8. 商家主账号可创建子账号，并按店铺/小程序分配维护权限。
9. 同一商家主体可维护多个小程序，且不同子账号可分别维护不同小程序。
10. 子账号权限应按角色和授权范围生效，未授权范围内的数据与操作不可见不可用。
11. 模板版本支持按商家发布、回滚和留痕。
12. 交付方可获得完整平台级源码包、部署文档和初始化脚本，并可在约定环境完成独立部署。
13. 满足标准前置条件的商家可通过一键流程完成开通和首版发布，且系统可输出完整校验与发布结果。
14. 商家可在商品管理页对多个商品执行一键发布，系统可明确返回成功和失败结果。
15. 对于预算敏感的小微企业，可采用轻量私有化部署方案完成首期上线，并保留后续平滑升配能力。

## 17. 风险与依赖

### 17.1 风险

- 商家定制需求边界不清，容易演变为持续分叉开发。
- 版本兼容性管理不到位，会导致不同商家发布失败或线上异常。
- 账单与退款规则复杂，容易引发财务对账问题。
- 多商家接入场景下，租户隔离和配置回退实现复杂。
- 源码交付后若缺少主干合并机制，容易形成私有化版本失控。

### 17.2 依赖

- 微信小程序接入能力
- 第三方支付通道
- 短信/邮件通知服务
- 物流接口服务
- 实名认证或企业信息校验服务
- 私有化部署环境和运维资源
