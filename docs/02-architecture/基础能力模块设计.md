# 程哒哒基础能力模块设计

## 1. 文档说明

- 文档用途：定义 `cdd-common-*` 的职责边界、首批公共能力清单和后续扩展约束。
- 当前阶段：基础能力设计基线 + 第一批代码骨架说明。
- 适用对象：后端研发、架构设计、测试、交付。

## 2. 设计目标

- 让各业务服务共享统一的错误模型、请求上下文、Web 出参、数据库审计字段和 Redis Key 规范。
- 避免每个服务重复实现一套响应结构、异常处理、请求头解析和基础工具类。
- 保证公共模块只沉淀横切能力，不反向承载任何具体业务领域对象和业务规则。

## 3. 总体原则

- `cdd-common-*` 只放横切能力，不放商家、商品、订单等领域模型。
- `cdd-common-core` 保持最轻依赖，尽量不引入重型 Spring 组件。
- 公共能力优先做“约定 + 骨架”，不在这一阶段提前塞满复杂实现。
- 任何公共能力都必须能服务至少两个以上业务模块，否则优先留在业务服务内。
- 公共模块禁止直接依赖 `cdd-api-*`、`cdd-*-service`、`cdd-gateway`、`cdd-pay-core`。

## 4. 模块职责定义

### 4.1 `cdd-common-core`

职责：

- 通用错误码模型
- 业务异常基类
- 分页基础模型
- 请求头常量
- 运行环境与配置模式枚举

不放内容：

- Spring MVC 组件
- 数据库访问实现
- Redis 客户端逻辑
- 认证鉴权实现

### 4.2 `cdd-common-web`

职责：

- 统一响应结构
- 分页响应结构
- 请求 ID 生成与透传
- 参数校验与业务异常的统一出参转换
- 面向 Spring MVC 的自动装配入口

不放内容：

- 业务 Controller
- 领域转换器
- 具体接口协议对象

### 4.3 `cdd-common-db`

职责：

- 数据库审计字段模型
- 通用列名约定
- MyBatis / JDBC 基础依赖承载

后续可继续扩展：

- MyBatis 自动填充
- 乐观锁与逻辑删除统一拦截
- 分页插件统一配置

不放内容：

- 业务表 `Mapper`
- 业务 SQL
- 跨域 Repository 实现

### 4.4 `cdd-common-redis`

职责：

- Redis Key 生成规范
- 分布式锁 Key 生成规范
- Redis 基础依赖承载

后续可继续扩展：

- RedisTemplate 序列化统一配置
- Redisson 接入基线
- 缓存穿透、空值缓存、TTL 约定

不放内容：

- 具体业务缓存 key 常量
- 商品、订单、营销等领域缓存封装

### 4.5 `cdd-common-security`

职责：

- 登录上下文基础模型
- 鉴权透传 Header 解析
- 安全过滤器链骨架
- 安全开关配置项

后续可继续扩展：

- JWT / Session 解析
- 权限点校验注解
- 商家、平台、消费者三端身份模型
- 数据范围与租户隔离扩展

不放内容：

- 具体账号登录流程
- 具体角色权限规则
- 业务审计落库逻辑

## 5. 第一批落地清单

当前建议先固定以下公共能力：

- 错误码：与接口文档中的 `40001`、`40101`、`50001` 等统一对齐
- 业务异常：所有服务统一抛 `BusinessException`
- 通用响应：统一输出 `code`、`message`、`data`、`request_id`
- 分页响应：统一输出 `list`、`page`、`page_size`、`total`
- 请求 ID：支持从 `X-Request-Id` 透传，没有则服务端生成
- 请求上下文 Header：先统一 `Authorization`、`X-Store-Id`、`X-Mini-Program-Id`、`X-Request-Id`
- DB 审计字段：统一 `created_at`、`updated_at`、`created_by`、`updated_by`、`deleted`、`version`
- Redis Key 规范：统一 `cdd:{env}:{domain}:{biz}:{identifier}`
- Redis 锁 Key 规范：统一 `cdd:lock:{domain}:{action}:{id}`
- 对外提示信息约束：统一响应 `message`、默认错误文案、业务异常提示统一使用中文，不使用英文默认提示

## 6. 首批不做的内容

- 不在当前阶段引入全局分布式事务框架
- 不在当前阶段做统一 RPC 框架封装
- 不在当前阶段抽象通用 Repository 基类
- 不在当前阶段做复杂权限树和注解鉴权
- 不在当前阶段把 MQ、对象存储、短信、微信能力塞进 `common-*`

说明：

- MQ、对象存储、支付、微信等属于“基础设施适配能力”，后续更适合拆独立模块或独立 starter，而不是直接塞进现有 `common-*`

## 7. 服务接入建议

- 所有可启动模块默认依赖：`cdd-common-core`、`cdd-common-web`
- 涉及数据库访问的服务再依赖：`cdd-common-db`
- 涉及缓存、分布式锁的服务再依赖：`cdd-common-redis`
- 需要登录态透传、安全过滤器骨架的服务再依赖：`cdd-common-security`

## 8. 后续演进顺序

建议按以下顺序继续往下做：

1. 先把 `common-web` 的统一响应、异常处理、请求 ID 在网关和业务服务里跑起来
2. 再把 `common-security` 的登录上下文透传接到网关和 `cdd-auth-service`
3. 再补 `common-db` 的审计字段自动填充与 MyBatis 统一配置
4. 最后补 `common-redis` 的序列化、缓存模板和分布式锁封装

## 9. 当前默认决策

- 公共错误码以接口设计文档为准
- 统一响应字段名采用 `snake_case` 对外输出
- 对外提示信息统一使用中文
- 请求追踪 ID 使用 `X-Request-Id`
- `common-*` 只沉淀横切能力，不沉淀业务规则
- 公共模块优先提供可复用骨架，不提前做复杂平台化封装
