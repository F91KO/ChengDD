# 程哒哒 Agent 成长路线图

## 1. 文档说明

- 文档用途：回答“如何让服务级 agent 持续成长，并逐步演进成可用于客户问题处理的机器人”。
- 适用对象：架构、研发、测试、实施、运维。
- 当前判断：先做“内部诊断与辅助处置”，再做“客户支持机器人”，不要一开始就追求完全自动修复。

## 2. 总体目标

- 让服务级 agent 从“会执行代码和测试”升级为“会收集上下文、会判断问题、会给出处理方案”。
- 让问题定位从依赖人工经验，逐步演进为依赖标准化数据、工具和案例库。
- 在低风险、可回放、可审计的前提下，逐步引入自动处置能力。

## 3. 核心原则

- 先做可观测，再做可判断，再做可处置。
- 先面向内部研发与运维，再面向客户。
- 所有自动动作必须可审计、可回放、可人工接管。
- 高风险动作必须保留审批与人工兜底。
- 对外提示、诊断结论、处理建议统一使用中文。

## 4. Agent 成长的五个支柱

### 4.1 服务边界知识

- 每个服务级 agent 都要有明确的职责边界。
- 要知道本服务的：
  - 核心模块
  - 关键配置
  - 依赖服务
  - 常见错误码
  - 常用验证命令

### 4.2 问题上下文采集

- 要能自动拿到定位问题所需的基础上下文：
  - 日志
  - traceId / requestId
  - 当前版本号
  - 最近发布记录
  - 配置快照
  - 数据库关键状态
  - 补偿任务状态

### 4.3 案例与 Runbook 沉淀

- 每次故障都应沉淀：
  - 现象
  - 影响范围
  - 根因
  - 修复动作
  - 回归方式
- 形成服务级案例库和标准 runbook。

### 4.4 工具化执行能力

- agent 不能只会“解释”，还要能“执行标准动作”。
- 典型工具包括：
  - 日志检索
  - 配置比对
  - 测试执行
  - 健康检查
  - 数据查询
  - 补偿任务重试建议

### 4.5 反馈闭环

- 每次诊断后都要评估：
  - 定位是否准确
  - 修复是否有效
  - 是否复发
  - 哪些知识缺失
- 将结果回写到知识库和 runbook 中。

## 5. 三阶段成长路径

### 5.1 一期：内部研发/测试诊断 Agent

目标：

- 先让 agent 会查、会汇总、会给诊断摘要。

主要能力：

- 读取服务日志和测试报告
- 区分编译失败、测试失败、环境失败、配置失败
- 给出失败摘要和复现命令
- 输出标准测试报告
- 识别影响模块和建议修复方向

典型使用者：

- 后端研发
- 测试
- 实施

验收标准：

- 能在一次执行中输出标准化诊断结果
- 能覆盖当前服务级编码 agent 的主要验证路径
- 能把失败原因压缩成可读的中文摘要

### 5.2 二期：内部运维处置 Agent

目标：

- 在诊断能力稳定后，引入低风险自动处置。

主要能力：

- 基础健康检查
- 配置巡检与差异提示
- 发布前检查
- 常见补偿任务巡检
- 可审批的低风险重试动作
- 对特定问题生成标准工单或处理建议

可优先自动化的动作：

- 重跑低风险验证
- 触发补偿任务重试建议
- 检查配置缺失
- 标记异常任务实例

仍需人工审批的动作：

- 生产配置变更
- 数据修复
- 强制重放业务任务
- 批量补偿执行

验收标准：

- 对低风险动作有明确边界
- 每个动作都有审计记录
- 失败时可回退到人工流程

### 5.3 三期：面向客户的支持机器人

目标：

- 面向商家、平台运营或客户支持人员，解决“客户看得懂的问题诊断与处理建议”。

主要能力：

- 按客户语言解释失败原因
- 根据订单号、商户号、任务号定位问题状态
- 给出下一步建议
- 自动生成工单或升级路径
- 对常见低风险问题提供半自动处理

适合优先支持的问题：

- 为什么发布失败
- 为什么商家开通卡住
- 为什么订单状态没有更新
- 为什么退款回调没有处理
- 为什么功能开关未生效

不建议一开始开放的能力：

- 直接修改生产数据
- 直接变更核心配置
- 不经审批直接执行高风险补偿

验收标准：

- 能输出客户可理解的解释
- 能明确区分“建议动作”和“已执行动作”
- 对外输出不暴露内部敏感实现细节

## 6. 能力演进顺序

建议按以下顺序建设：

1. 服务级 agent 边界与职责清单
2. 错误码体系与中文提示统一
3. 测试报告、失败摘要、复现命令统一
4. 日志、traceId、配置快照接入
5. 服务级 runbook 和案例库
6. 低风险诊断工具统一接入
7. 低风险自动处置
8. 面向客户的问答与诊断机器人

## 7. 当前项目需要补齐的基础设施

### 7.1 可观测性

- 全链路 `traceId` / `requestId`
- 服务级统一日志格式
- 发布记录与版本记录
- 配置变更审计
- 补偿任务执行状态可见

### 7.2 知识体系

- 服务级问题案例库
- 错误码说明文档
- 常见故障 runbook
- 测试失败样例库

### 7.3 工具体系

- 服务级健康检查脚本
- 测试执行脚本
- 配置比对工具
- 日志检索入口
- 补偿任务诊断工具

## 8. 服务级 Agent 的成长方向

### 8.1 `auth-service` agent

- 成长方向：
  - 登录失败诊断
  - token 失效原因分析
  - 刷新链路异常定位

### 8.2 `gateway` agent

- 成长方向：
  - 鉴权失败诊断
  - 上下文透传缺失定位
  - 请求路由异常分析

### 8.3 `merchant-service` agent

- 成长方向：
  - 商家开通任务卡点分析
  - 小程序接入参数校验失败定位
  - 入驻申请状态异常诊断

### 8.4 `product-service` agent

- 成长方向：
  - 分类模板初始化问题定位
  - 商品上下架失败分析
  - 库存调整异常诊断

### 8.5 `order-service` agent

- 成长方向：
  - 下单链路异常定位
  - 支付回调幂等问题分析
  - 退款补偿链路诊断

### 8.6 `release-service` agent

- 成长方向：
  - 发布失败定位
  - 回滚任务诊断
  - 版本映射异常分析

### 8.7 `config-service` agent

- 成长方向：
  - 配置未生效诊断
  - 商家级覆盖冲突分析
  - 功能开关异常定位

### 8.8 `common-security` agent

- 成长方向：
  - 角色与数据权限拦截诊断
  - 权限不足与范围错误分析

### 8.9 `agent-core` agent

- 成长方向：
  - 任务状态卡住诊断
  - 步骤执行失败聚合分析
  - 补偿链路恢复建议

## 9. 机器人化落地建议

### 9.1 第一层：问答机器人

- 只回答“是什么、为什么、怎么查”。
- 不执行生产动作。

### 9.2 第二层：诊断机器人

- 读取上下文后输出高概率根因。
- 生成处理建议或工单建议。

### 9.3 第三层：半自动处理机器人

- 对低风险、标准化问题执行预定义动作。
- 所有动作保留审计记录。

### 9.4 第四层：自动处置机器人

- 只适用于低风险、可回滚、规则明确的动作。
- 必须有审批机制和熔断机制。

## 10. 当前建议的落地策略

- 当前阶段优先做“一期内部研发/测试诊断 agent”。
- 第二优先级做“内部运维处置 agent”。
- 面向客户的机器人放在第三阶段，不建议现在直接做。
- 每个阶段先从一个问题最集中、链路最清晰的服务开始，优先推荐：
  - `order-service`
  - `merchant-service`
  - `release-service`
