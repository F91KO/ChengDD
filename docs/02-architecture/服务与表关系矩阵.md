# 程哒哒服务与表关系矩阵

## 1. 文档说明

- 文档用途：明确各服务负责的数据表范围、主写责任和跨服务访问边界。
- 当前阶段：数据库设计前的服务归属矩阵，不等同于最终建表 SQL。
- 使用场景：技术评审、数据库设计、接口设计、拆库规划。

## 2. 设计原则

- 表命名统一采用 `cdd_{domain}_{object}`。
- 每张核心业务表必须明确唯一主责任服务。
- 跨服务不直接写其他服务主表。
- 跨服务读取优先通过接口、事件同步或只读汇总表实现。
- 一期采用单库多表，但逻辑上严格按服务边界分域。

## 3. 服务与表关系总览

| 服务 | 领域职责 | 主责表前缀 | 是否允许被其他服务直接写入 |
| --- | --- | --- | --- |
| `cdd-merchant-service` | 商家、店铺、子账号、入驻接入 | `cdd_merchant_*` / `cdd_store_*` | 否 |
| `cdd-decoration-service` | 首页配置、装修版本、首页模块 | `cdd_decoration_*` | 否 |
| `cdd-product-service` | 分类模板、分类属性、商品、SKU、库存 | `cdd_product_*` | 否 |
| `cdd-order-service` | 购物车、结算、订单、支付结果、售后 | `cdd_order_*` | 否 |
| `cdd-marketing-service` | 优惠券、活动、首页营销会场、推荐规则 | `cdd_marketing_*` | 否 |
| `cdd-release-service` | 模板版本、发布任务、灰度、回滚、发布日志 | `cdd_release_*` / `cdd_template_*` | 否 |
| `cdd-report-service` | 报表、统计快照、埋点汇总 | `cdd_report_*` / `cdd_metric_*` | 否 |
| `cdd-auth-service` | 平台账号、登录态、角色权限、审计 | `cdd_auth_*` / `cdd_audit_*` | 否 |
| `cdd-config-service` | 配置中心、功能开关、预置规则 | `cdd_config_*` | 否 |

## 12. 跨服务访问规则

| 场景 | 推荐方式 | 不推荐方式 |
| --- | --- | --- |
| 订单查询商品可售状态 | 调 `cdd-product-service` 接口 | 直接查 `cdd_product_*` 表 |
| 首页读取活动会场 | 调 `cdd-marketing-service` 接口 | 装修服务直接写营销表 |
| 开通初始化分类 | `cdd-release-service` 调 `cdd-product-service` | 发布服务直接写分类表 |
| 报表统计订单数据 | 订阅事件或汇总同步 | 报表服务频繁直查订单主表 |
| 权限校验 | 调 `cdd-auth-service` | 各服务自行维护一套权限表 |

## 13. 一期建议的拆库预留

- 保持单库多表，但表名前缀严格按服务域划分。
- 后续若要拆库，优先候选：
  - 商品域：`cdd_product_*`
  - 订单域：`cdd_order_*`
  - 发布域：`cdd_release_*` / `cdd_template_*`
  - 报表域：`cdd_report_*` / `cdd_metric_*`
- 即使物理未拆库，代码层也必须只访问本服务主责表。
