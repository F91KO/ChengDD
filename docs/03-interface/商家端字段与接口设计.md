# 程哒哒商家端字段与接口设计

## 1. 文档说明

- 文档目标：基于 PRD 和商家端功能细化文档，补充商家端主要模块的字段设计、接口清单和关键校验规则。
- 适用对象：产品经理、后端研发、前端研发、测试、实施。
- 当前阶段：接口与对象设计草案，用于原型深化、接口评审和研发排期，不等同于最终数据库建表文档。

## 2. 设计原则

1. 商家端接口默认在商家登录态下访问，商家边界由登录上下文自动注入，不允许前端随意指定其他商家 `merchant_id`。
2. 商家可维护多个店铺和多个小程序，接口需明确 `store_id` 或 `mini_program_id` 作用范围。
3. 所有发布类、批量类、高风险类接口必须记录操作人、操作时间和操作结果。
4. 关键流程优先返回“可执行结果 + 阻塞项 + 提示项”，方便前端引导商家修复问题。
5. 字段命名在接口层统一采用 `snake_case` 风格。

## 3. 通用接口约定

### 3.1 请求头

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `Authorization` | 是 | 商家登录令牌，格式 `Bearer {token}` |
| `X-Store-Id` | 否 | 当前操作店铺 ID；店铺级接口建议显式传入 |
| `X-Mini-Program-Id` | 否 | 当前操作小程序 ID；发布和装修相关接口建议显式传入 |
| `X-Request-Id` | 否 | 请求追踪 ID |

### 3.2 通用响应结构

```json
{
  "code": 0,
  "message": "成功",
  "data": {},
  "request_id": "req_20260320_xxx"
}
```

### 3.3 分页结构

```json
{
  "list": [],
  "page": 1,
  "page_size": 20,
  "total": 100
}
```

### 3.4 常见错误码建议

| 错误码 | 说明 |
| --- | --- |
| `40001` | 参数错误 |
| `40003` | 状态不允许当前操作 |
| `40004` | 必填字段缺失 |
| `40009` | 校验未通过 |
| `40101` | 未登录或令牌失效 |
| `40301` | 无权限访问 |
| `40401` | 资源不存在 |
| `40901` | 数据冲突 |
| `50001` | 系统异常 |

## 6. 关键接口详细说明

### 6.1 提交入驻申请

接口：
- `POST /merchant/applications/submit`

请求体：

```json
{
  "merchant_name": "程哒哒鲜选",
  "merchant_type": "enterprise",
  "contact_name": "张三",
  "contact_mobile": "13800000000",
  "legal_person_name": "张三",
  "business_category": "fresh_food",
  "brand_name": "鲜选优品",
  "license_file_url": "https://xx/license.png",
  "legal_id_front_url": "https://xx/id-front.png",
  "legal_id_back_url": "https://xx/id-back.png",
  "trademark_file_url": "https://xx/tm.png"
}
```

校验规则：
- `merchant_name`、`merchant_type`、`contact_name`、`contact_mobile`、`legal_person_name`、`business_category` 为必填。
- 文件字段必须为已上传成功的文件地址。
- 已提交且处于待处理状态时，不允许重复提交新申请，只允许补充资料。

响应体：

```json
{
  "code": 0,
  "message": "成功",
  "data": {
    "application_id": "app_1001",
    "status": "submitted"
  }
}
```

### 6.2 一键接入检测

接口：
- `POST /merchant/mini-programs/{mini_program_id}/detect`

响应字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `overall_passed` | bool | 是否整体通过 |
| `blocking_items` | array | 阻塞项 |
| `warning_items` | array | 提示项 |
| `last_detected_at` | datetime | 最近检测时间 |

阻塞项示例：

```json
{
  "field": "payment_mch_id",
  "reason": "支付商户号缺失",
  "suggestion": "请补充微信支付商户号后重试"
}
```

### 6.4 创建子账号

接口：

请求体：

```json
{
  "account_name": "ops_beijing",

  "mobile": "13900000000",
  "remark": "负责北京店铺",
```

### 6.5 新增或更新分类

接口：
- `POST /merchant/categories`
- `PUT /merchant/categories/{category_id}`

关键字段：
- `category_name`
- `parent_id`
- `sort_order`
- `icon_url`
- `description`
- `is_enabled`
- `is_visible`
- `attributes`

规则：
- 分类层级不得超过平台设定上限。
- 分类名称不得命中禁用词。
- 停用分类后，前台入口隐藏，但历史订单分类快照保留。

### 6.6 维护分类属性

接口：
- `POST /merchant/categories/{category_id}/attributes`
- `PUT /merchant/categories/attributes/{attribute_id}`

请求体：

```json
{
  "attribute_name": "糖度",
  "attribute_code": "sugar_level",
  "attribute_type": "single_select",
  "is_required": true,
  "is_filterable": true,
  "is_visible": true,
  "supports_multiple": false,
  "option_values": ["高", "中", "低"],
  "sort_order": 10
}
```

规则：
- `attribute_name`、`attribute_type` 必填。
- 同一分类下属性名称不可重复。
- 若属性已被商品使用，删除前必须提示先处理历史数据或依赖商品属性快照保留历史值。
- 受控分类的必备属性不得由商家直接删除。

```json
  "category_id": "cat_1001",
  "product_name": "精品苹果礼盒",
```

接口：
- `POST /merchant/after-sales/{after_sale_id}/audit`

## 7. 前后端联调重点

1. 商家边界不要放到前端可修改参数中，应以后端登录态识别为准。
2. 草稿保存和正式提交必须分接口或分状态，避免误提交。
3. 批量发布、一键开通、装修发布都必须返回可展示给商家的失败原因。
4. 文件上传接口与业务提交接口要解耦，避免一次请求体过大。
5. 商品、分类、分类属性、开关、装修配置都应支持版本或变更留痕。
