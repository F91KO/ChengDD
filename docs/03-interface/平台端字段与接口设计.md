# 程哒哒平台端字段与接口设计

## 1. 文档说明

- 文档目标：基于 PRD 和平台端功能细化文档，补充平台端主要模块的字段设计、接口清单和关键治理规则。
- 适用对象：产品经理、后端研发、前端研发、测试、交付实施、运维。
- 当前阶段：接口与对象设计草案，用于后台原型、接口评审、研发拆分和交付实施准备。

## 2. 设计原则

1. 平台端接口默认在平台账号登录态下访问，必须区分平台交付管理员与平台管理员。
2. 平台端所有涉及商家操作的接口，都必须显式带 `merchant_id` 或目标范围。
3. 发布、冻结、回滚、授权、扩展审批、CI/CD 执行等高风险接口必须留痕。
4. 平台端优先输出“治理结果 + 风险提示 + 失败原因 + 建议动作”，方便交付人员处理问题。
5. 字段命名统一采用 `snake_case` 风格。

## 3. 通用接口约定

### 3.1 请求头

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `Authorization` | 是 | 平台登录令牌，格式 `Bearer {token}` |
| `X-Request-Id` | 否 | 请求追踪 ID |
| `X-Operator-Id` | 否 | 当前操作人 ID，网关可自动注入 |

### 3.2 通用响应结构

```json
{
  "code": 0,
  "message": "成功",
  "data": {},
  "request_id": "req_20260320_xxx"
}
```

### 3.3 常见错误码建议

| 错误码 | 说明 |
| --- | --- |
| `40001` | 参数错误 |
| `40003` | 当前状态不允许操作 |
| `40009` | 兼容性或治理校验未通过 |
| `40101` | 未登录或令牌失效 |
| `40301` | 无权限访问 |
| `40401` | 资源不存在 |
| `40901` | 数据冲突 |
| `50001` | 系统异常 |

## 6. 关键接口详细说明

### 6.1 处理商家申请

接口：
- `POST /platform/merchants/{merchant_id}/process`

请求体：

```json
{
  "action": "approve",
  "reason": ""
}
```

字段说明：
- `action` 取值：`approve` / `reject` / `require_supplement`
- `reason` 在 `reject` 和 `require_supplement` 时必填

响应体：

```json
{
  "code": 0,
  "message": "成功",
  "data": {
    "merchant_id": "m_1001",
    "application_status": "approved"
  }
}
```

规则：
- 仅 `submitted` / `pending_process` / `supplement_required` 状态允许处理。
- 处理动作必须写入处理记录和审计日志。

### 6.2 创建分类模板

接口：
- `POST /platform/category-templates`

请求体：

```json
{
  "template_name": "生鲜零售模板",
  "industry_code": "fresh_food",
  "template_version": "v1.0.0",
  "max_level": 3,
  "default_sort_rule": "manual",
  "qualification_requirements": ["food_license"]
}
```

规则：
- `template_name`、`industry_code`、`template_version`、`max_level` 必填。
- 同一行业下模板名称 + 版本号组合不可重复。

### 6.3 维护分类属性模板

接口：
- `POST /platform/category-templates/{template_id}/attribute-templates`
- `PUT /platform/category-templates/attribute-templates/{template_attribute_id}`

请求体：

```json

  "node_id": "node_fruit_01",
  "attribute_name": "糖度",
  "attribute_code": "sugar_level",
  "attribute_type": "single_select",
  "is_required": true,
  "supports_multiple": false,
  "is_controlled": false,
  "option_values": ["高", "中", "低"],
  "sort_order": 10
}
```

规则：

- 同一模板节点下 `attribute_code` 必须唯一。
- 已被大量商家采用的属性模板变更时，应提示影响范围并建议以新增版本方式处理。

## 7. 关键治理规则

### 7.1 商家治理

- 商家处理通过后，才允许进入标准开通和发布链路。
- 冻结商家后，不允许新发版，但需保留数据查看和历史记录。

### 7.2 发布治理

- 发布前必须先校验，再执行发布。
- 平台需支持按商家灰度、按商家回滚、按失败阶段提示原因。
- 回滚能力默认仅主管理员或具备授权的发布管理员可执行。

### 7.3 模板与扩展治理

- 模板升级不得直接覆盖商家生效配置。
- 分类属性模板仅作为初始化来源，不得直接覆盖商家已生效的分类属性。
- 扩展组件必须声明兼容版本、风险等级和降级方案。
- 高风险扩展必须走审批流程。

### 7.4 权限与审计治理

- 权限必须做到菜单、动作、数据范围分离。
- 审计日志至少覆盖：商家处理、模板发布、开关变更、回滚、冻结、CI/CD 执行。

### 7.5 CI/CD 治理

- CI/CD 服务是可选交付项，不作为轻量版首期强制条件。
- 流水线配置、执行记录、回滚记录必须可追踪。
- 若客户为私有化部署，平台提供 CI/CD 服务时需明确仓库权限、执行环境和责任边界。

## 8. 联调重点

1. 平台侧列表查询要支持高频筛选条件，避免只给简单分页。
2. 兼容性校验结果要结构化，不能只返回字符串。
3. 发布失败原因要和前端页面可展示字段一一对应。
4. 所有高风险接口都要输出审计日志 ID 或可追踪标识。
5. CI/CD 接口要区分“流水线配置”和“流水线执行”，避免耦合。
