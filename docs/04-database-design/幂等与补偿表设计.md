# 程哒哒幂等与补偿表设计

## 1. 文档说明

- 本文用于补足支付、退款、发布、开通初始化等高风险链路的幂等与补偿落库设计。
- 目标不是替代业务主表，而是把“重复请求识别、回调留痕、失败重放、人工补偿”独立出来。
- 本文与 [状态流转矩阵与SQL落库说明](./状态流转矩阵与SQL落库说明.md) 配套使用。

## 2. 设计目标

### 2.1 必须解决的问题

1. 第三方回调重复投递，不能重复改主状态。
2. 本地事务成功后，外围动作失败，需要可重试。
3. 跨服务编排部分成功，后续需要人工或系统补偿。
4. 重试过程必须可审计、可追踪、可限流。

### 2.2 不在本轮解决的问题

- 全链路消息总线统一去重。
- 通用分布式事务框架。
- 跨机房或跨区域容灾补偿。

## 3. 总体设计原则

### 3.1 业务主表不承担完整幂等职责

- 业务主表上的唯一键只能解决一部分重复写问题。
- 真正的幂等需要单独记录“收到过什么请求、处理到了哪一步、结果是什么”。

### 3.2 幂等记录与业务状态解耦

- 回调请求先落幂等记录，再决定是否更新业务主表。
- 业务主表只保留业务状态，不保存复杂的重复请求处理过程。

### 3.3 补偿任务单独建模

- 补偿不是简单重试。
- 补偿任务要能记录：
  - 为什么需要补偿
  - 重试了多少次
  - 当前卡在哪一步
  - 最终是成功、失败还是转人工

## 4. 一期建议新增的表

| 表名 | 用途 | 归属域 |
| --- | --- | --- |
| `cdd_order_pay_callback_record` | 支付回调幂等记录 | 订单域 |
| `cdd_order_refund_callback_record` | 退款回调幂等记录 | 订单域 |
| `cdd_release_callback_record` | 发布平台回调/异步结果幂等记录 | 发布域 |
| `cdd_compensation_task` | 通用补偿任务主表 | 编排/平台域 |

## 5. 支付回调幂等设计

### 5.1 幂等键

优先级：

1. `third_party_trade_no`
2. `pay_no`
3. `merchant_id + pay_channel + callback_event_id`

### 5.2 表设计要点

表：`cdd_order_pay_callback_record`

核心字段：
- `id`
- `pay_record_id`
- `pay_no`
- `merchant_id`
- `store_id`
- `pay_channel`
- `third_party_trade_no`
- `callback_event_id`
- `callback_status`
- `callback_payload_json`
- `processed_result`
- `processed_at`

状态建议：
- `received`
- `processing`
- `processed`
- `ignored`
- `failed`

落库流程：

1. 先按幂等键插入或查询 `cdd_order_pay_callback_record`
2. 若已是 `processed`，直接返回成功
3. 若为首次处理，更新为 `processing`
4. 在本地事务中更新 `cdd_order_pay_record`、`cdd_order_info`、`cdd_order_status_log`
5. 成功后把回调记录置为 `processed`
6. 失败则置为 `failed`，并视情况创建补偿任务

### 5.3 为什么不能只靠 `uk_third_party_trade_no_deleted`

- 唯一索引只能防止重复写入同一第三方单号。
- 但无法记录“本次回调为什么被忽略、是否已完整处理、失败原因是什么”。
- 也无法承接第三方没有单号或单号为空的异常情况。

## 6. 退款回调幂等设计

### 6.1 幂等键

优先级：

1. `third_party_refund_no`
2. `refund_no`
3. `merchant_id + callback_event_id`

### 6.2 表设计要点

表：`cdd_order_refund_callback_record`

核心字段：
- `id`
- `refund_record_id`
- `refund_no`
- `order_id`
- `merchant_id`
- `store_id`
- `third_party_refund_no`
- `callback_event_id`
- `callback_status`
- `callback_payload_json`
- `processed_result`
- `processed_at`

状态建议：
- `received`
- `processing`
- `processed`
- `ignored`
- `failed`

落库流程：

1. 回调请求先写 `cdd_order_refund_callback_record`
2. 若同一退款已成功处理，则直接返回成功
3. 本地事务内更新：
   - `cdd_order_refund_record.refund_status`
   - `cdd_order_after_sale.after_sale_status`
   - `cdd_order_info.pay_status`
4. 若主事务失败，记录 `failure_reason` 并创建补偿任务

## 7. 发布回调幂等设计

### 7.1 场景说明

发布链路存在多次异步回调：
- 平台提审结果
- 审核通过/驳回结果
- 代码包提交结果
- 灰度完成结果

### 7.2 幂等键

建议主键组合：

- `task_id + step_code + callback_event_id`

若第三方平台提供全局事件号，可优先使用：

- `source_system + callback_event_id`

### 7.3 表设计要点

表：`cdd_release_callback_record`

核心字段：
- `id`
- `task_id`
- `task_no`
- `merchant_id`
- `store_id`
- `mini_program_id`
- `step_code`
- `source_system`
- `callback_event_id`
- `callback_status`
- `callback_payload_json`
- `processed_result`
- `processed_at`

状态建议：
- `received`
- `processing`
- `processed`
- `ignored`
- `failed`

落库流程：

1. 先写或查回调记录
2. 仅允许一个处理线程把状态改为 `processing`
3. 更新对应 `cdd_release_task_detail.step_status`
4. 必要时推进 `cdd_release_task.release_status`
5. 写 `cdd_release_log`
6. 成功后置 `processed`

## 8. 补偿任务设计

### 8.1 为什么需要通用补偿任务表

以下场景都不适合只靠定时扫描主表解决：

- 支付回调已成功入库，但订单状态回写失败
- 退款状态成功，但售后单未同步完成
- 发布任务成功，但 `current_template_version` 未回写
- 开通初始化时分类已生成，但装修初始化失败

### 8.2 表设计要点

表：`cdd_compensation_task`

核心字段：
- `id`
- `task_code`
- `biz_type`
- `biz_id`
- `compensation_type`
- `task_status`
- `retry_count`
- `max_retry_count`
- `next_retry_at`
- `last_error_code`
- `last_error_message`
- `payload_json`
- `resolved_at`

状态建议：
- `pending`
- `running`
- `success`
- `failed`
- `dead`
- `cancelled`

### 8.3 使用规则

- 同一业务对象同一补偿类型只保留一条活跃任务。
- `task_code` 全局唯一，便于排查。
- `payload_json` 保存最小重放上下文，不复制全部大对象。
- 达到 `max_retry_count` 后转 `dead`，等待人工处理。

## 9. 建议的 DDL 约束

### 9.1 `cdd_order_pay_callback_record`

建议唯一约束：
- `uk_pay_no_event_deleted(pay_no, callback_event_id, deleted)`
- `uk_trade_no_deleted(third_party_trade_no, deleted)`

建议索引：
- `idx_pay_record_id(pay_record_id)`
- `idx_status_next_time(callback_status, created_at)`

### 9.2 `cdd_order_refund_callback_record`

建议唯一约束：
- `uk_refund_no_event_deleted(refund_no, callback_event_id, deleted)`
- `uk_third_party_refund_no_deleted(third_party_refund_no, deleted)`

建议索引：
- `idx_refund_record_id(refund_record_id)`
- `idx_status_created(callback_status, created_at)`

### 9.3 `cdd_release_callback_record`

建议唯一约束：
- `uk_task_step_event_deleted(task_id, step_code, callback_event_id, deleted)`

建议索引：
- `idx_task_step(task_id, step_code)`
- `idx_status_created(callback_status, created_at)`

### 9.4 `cdd_compensation_task`

建议唯一约束：
- `uk_task_code_deleted(task_code, deleted)`
- `uk_biz_comp_type_active_deleted(biz_type, biz_id, compensation_type, deleted)`

建议索引：
- `idx_status_next_retry(task_status, next_retry_at)`
- `idx_biz_type_id(biz_type, biz_id)`
