# 程哒哒数据库设计实施细化

## 1. 文档定位

- 本文是对 [数据库设计草案](./数据库设计草案.md) 和 [schema-draft.sql](./schema-draft.sql) 的落地补充。
- 目标不是重新定义业务范围，而是把“逻辑草案”推进到“可收口为正式 DDL”的程度。
- 当前优先服务一期闭环：认证与权限、商家入驻、小程序接入、一键开通、分类商品、订单支付、发布编排。

## 2. 一期数据库目标

### 2.1 一期必须覆盖的业务闭环

1. 平台账号登录并处理商家入驻申请。
2. 商家完成主体资料、店铺资料、小程序接入。
3. 商家初始化首页模板、分类、商品并执行首版发布。
4. 消费者完成下单、支付、履约、退款/售后。
5. 平台和商家都能查询发布结果、订单结果、审计日志。

### 2.2 一期主表范围

| 业务链路 | 必须落库的核心表 |
| --- | --- |
| 认证权限 | `cdd_auth_account`、`cdd_auth_role`、`cdd_auth_permission`、`cdd_auth_role_permission`、`cdd_audit_log` |
| 商家接入 | `cdd_merchant_application`、`cdd_merchant_profile`、`cdd_merchant_store`、`cdd_merchant_mini_program` |
| 商家账号 | `cdd_merchant_sub_account`、`cdd_merchant_account_scope` |
| 店铺装修 | `cdd_decoration_home_config`、`cdd_decoration_home_module`、`cdd_decoration_home_version`、`cdd_decoration_template_binding` |
| 商品库存 | `cdd_product_category_template`、`cdd_product_category_template_attribute`、`cdd_product_category`、`cdd_product_category_attribute`、`cdd_product_spu`、`cdd_product_sku`、`cdd_product_stock`、`cdd_product_publish_record` |
| 交易链路 | `cdd_order_cart_item`、`cdd_order_checkout_snapshot`、`cdd_order_info`、`cdd_order_item`、`cdd_order_pay_record`、`cdd_order_refund_record`、`cdd_order_after_sale`、`cdd_order_status_log` |
| 发布编排 | `cdd_template_version`、`cdd_template_change_log`、`cdd_release_task`、`cdd_release_task_detail`、`cdd_release_gray_record`、`cdd_release_rollback_record`、`cdd_release_log` |
| 配置发号 | `cdd_config_feature_switch`、`cdd_config_feature_switch_value`、`cdd_config_preset_rule`、`cdd_config_kv`、`cdd_config_id_segment` |

### 2.3 一期可延后优化项

- 复杂分库分表。
- 统一枚举字典表。
- 通用操作日志大宽表归档体系。
- 报表宽表的准实时化和独立分析库。
- 面向搜索场景的 ES 或 OpenSearch。

## 3. 多租户与数据边界细化

### 3.1 租户主键定义

- `merchant_id` 是一级租户主键，商家侧核心数据必须可按 `merchant_id` 隔离。
- `store_id` 是二级经营实体主键，交易、商品、装修、发布明细必须尽量带上 `store_id`。
- `mini_program_id` 是渠道侧主键，用于接入和发布绑定，不替代 `store_id`。

### 3.2 哪些表必须带租户字段

| 表类型 | 要求 |
| --- | --- |
| 商家私有业务表 | 必须带 `merchant_id` |
| 店铺经营表 | 必须带 `merchant_id` 和 `store_id` |
| 小程序接入/发布表 | 必须带 `merchant_id`，涉及具体发布对象时带 `store_id`、`mini_program_id` |
| 平台公共模板表 | 可不带 `merchant_id`，但需要明确平台态 `status` |
| 认证平台表 | 平台账号表可不带 `merchant_id`，商家子账号与权限范围表必须能反查商家归属 |

### 3.3 查询与更新约束

- 商家后台所有列表查询默认必须附带 `merchant_id` 条件。
- 交易、商品、装修页的详情查询必须同时校验 `merchant_id` 与对象 `id`。
- 店铺维度页面推荐统一走 `merchant_id + store_id + id` 三段校验，避免横向越权。
- 平台侧跨商家查询允许不带 `merchant_id`，但要带角色与数据权限校验。

## 4. 通用字段与类型细化

### 4.1 主键与业务单号

- 内部关联统一使用 `id bigint`。
- 对外展示和排查统一使用业务编码字段，例如 `merchant_no`、`order_no`、`task_no`。
- 主键发号建议继续采用 `cdd_config_id_segment` 号段模式，不引入数据库自增。

### 4.2 金额字段

- 一期沿用 `decimal(10,2)`，与现有 `schema-draft.sql` 保持一致。
- 所有金额字段默认非负，除非语义上允许冲正。
- 金额命名保持统一：
  - 原价或总额：`total_amount`
  - 优惠额：`discount_amount`
  - 应付额：`payable_amount`
  - 实付额：`paid_amount`
  - 退款额：`refund_amount`
- 计算规则必须在应用层统一，数据库不通过触发器维护金额联动。

### 4.3 状态字段

- 状态字段统一采用 `varchar(32)`，避免早期 enum 变更成本。
- 同一表只保留能稳定表达业务阶段的 1 到 3 个状态字段，不叠加过多派生状态。
- 状态变更必须可追溯：
  - 订单走 `cdd_order_status_log`
  - 发布走 `cdd_release_log`
  - 审计动作走 `cdd_audit_log`

### 4.4 布尔与开关字段

- 布尔字段统一使用 `tinyint(1)`，命名为 `is_*`、`has_*`、`can_*`。
- 开关配置不直接散落在业务表中，优先落到 `cdd_config_feature_switch` 与 `cdd_config_feature_switch_value`。

### 4.5 JSON 字段使用边界

- 仅以下场景允许使用 JSON：
  - 第三方回包或检测结果
  - 快照数据
  - 装修模块配置
  - 候选值、规则配置、扩展字段
- JSON 字段不能承担核心查询条件。
- 若某 JSON 内字段开始进入高频筛选，必须拆为显式列并补索引。
- 建议所有 JSON 结构内包含 `schema_version` 字段，便于后续兼容演进。

### 4.6 审计与软删除

- 所有核心表保留 `deleted` 和 `version` 字段。
- 软删除只用于业务侧“撤销可恢复”的对象，不用于高频流水清理。
- 流水日志类表原则上不执行软删除业务操作，只保留框架公共字段以兼容统一基础设施。

## 5. 一期核心关系与落库顺序

### 5.1 商家接入链路

关系说明：
- `cdd_merchant_application` 记录申请过程。
- 审核通过后生成 `cdd_merchant_profile`。
- 商家主体下创建 `cdd_merchant_store`。
- 店铺接入小程序后生成 `cdd_merchant_mini_program`。

建议约束：
- 一个 `merchant_profile` 同时只关联一个 `current_application_id`。
- `cdd_merchant_store` 以 `merchant_id + store_name + deleted` 保证商家内店铺名唯一。
- `cdd_merchant_mini_program` 以 `app_id + deleted` 保证全局唯一接入。

### 5.2 商品与库存链路

关系说明：
- 平台模板：`cdd_product_category_template` -> `cdd_product_category_template_attribute`
- 商家分类：`cdd_product_category` -> `cdd_product_category_attribute`
- 商品主数据：`cdd_product_spu` -> `cdd_product_sku` -> `cdd_product_stock`
- 发布结果：`cdd_product_publish_record`

建议约束：
- `cdd_product_stock` 对 `sku_id` 做唯一约束，保持“一 SKU 一库存主记录”。
- `cdd_product_sku` 不建议复用软删除后的 `sku_code`，避免外部渠道排查混乱。
- 分类模板初始化到商家分类时，保留 `template_id` 追溯来源。

### 5.3 订单交易链路

关系说明：
- `cdd_order_cart_item` 仅保存购物车态。
- `cdd_order_checkout_snapshot` 固化提交时的商品、优惠、金额快照。
- `cdd_order_info` 保存订单主状态。
- `cdd_order_item` 保存商品明细快照。
- `cdd_order_pay_record` 与 `cdd_order_refund_record` 承接资金流水。
- `cdd_order_after_sale` 承接售后主流程。

建议约束：
- 订单创建后，商品名称、SKU 名称、单价必须写入快照，不依赖商品表回查。
- 支付回调、退款回调必须基于业务单号和第三方单号双重幂等控制。
- 售后与退款要允许一单多次售后，但需要明确按订单项还是按订单粒度处理。

### 5.4 发布编排链路

关系说明：
- `cdd_template_version` 维护模板版本主记录。
- `cdd_release_task` 维护一次发布任务。
- `cdd_release_task_detail` 维护具体门店/小程序粒度执行结果。
- `cdd_release_gray_record` 与 `cdd_release_rollback_record` 维护灰度与回滚附属记录。

建议约束：
- `task_no` 必须全局唯一。
- 平台统一发布走单任务多明细。
- 商家首版开通也复用发布任务体系，不单独建另一套流程表。

## 7. 事务与一致性规则

### 7.1 适合本地事务的场景

| 场景 | 涉及表 | 规则 |
| --- | --- | --- |
| 商家申请提交 | `cdd_merchant_application` | 单表写，直接提交 |
| 审核通过开通商家 | `cdd_merchant_application`、`cdd_merchant_profile`、`cdd_merchant_store` | 一个本地事务内完成审核结果和主数据初始化 |
| 商品保存 | `cdd_product_spu`、`cdd_product_sku`、`cdd_product_stock` | 以商品为聚合根，本地事务提交 |
| 订单提交 | `cdd_order_checkout_snapshot`、`cdd_order_info`、`cdd_order_item` | 订单主从表一事务落库 |
| 支付结果回写 | `cdd_order_pay_record`、`cdd_order_info`、`cdd_order_status_log` | 基于支付回调做幂等更新 |

### 7.2 不适合跨服务强事务的场景

- 发布任务编排不做跨服务分布式事务，由 `cdd_release_task` 记录阶段结果。
- 库存锁定与订单创建之间采用“订单服务发起，商品服务确认”的接口/事件组合，不直接共享事务。
- 报表更新不阻塞交易主链路，采用异步汇总。

### 7.3 幂等与并发控制

- 支付回调幂等键：`pay_no` 或 `third_party_trade_no`。
- 退款回调幂等键：`refund_no` 或 `third_party_refund_no`。
- 发布回调幂等键：`task_id + detail_id + stage_code`。
- 库存扣减依赖 `version` 乐观锁或条件更新：
  - `available_stock >= 扣减量`
  - 成功后同步增加 `locked_stock` 或减少 `available_stock`

## 8. 索引与查询模式细化

### 8.1 一期高频查询模式

| 场景 | 推荐索引 |
| --- | --- |
| 平台按状态查看入驻申请 | `status + submitted_at` |
| 商家查看门店商品列表 | `store_id + category_id + status` |
| 商家查看订单列表 | `store_id + order_status + created_at` |
| 消费者查看我的订单 | `user_id + created_at` |
| 支付回调查流水 | `pay_no`、`third_party_trade_no` |
| 平台查看发布任务 | `merchant_id + release_status` |
| 店铺查看售后列表 | `store_id + after_sale_status` |

### 8.2 索引控制原则

- 不单独给低区分度字段如 `status` 建大量索引。
- 联合索引优先覆盖“租户字段 + 状态字段 + 时间字段”。
- `text`、`json` 字段不参与 B-Tree 核心索引设计。
- 上线前需要用真实分页 SQL 复核索引是否命中，而不是只看字段存在。

## 9. JSON 字段治理建议

### 9.1 快照类

- `cdd_order_checkout_snapshot.product_snapshot_json`
- `cdd_order_checkout_snapshot.coupon_snapshot_json`
- `cdd_order_item.sku_spec_json`
- `cdd_decoration_home_version.snapshot_json`

规则：
- 必须包含生成时间和版本号。
- 快照生成后不回写修正，只新增版本。

### 9.2 配置类

- `cdd_decoration_home_module.config_payload_json`
- `cdd_marketing_activity.config_json`
- `cdd_config_preset_rule.rule_payload_json`

规则：
- 必须定义最外层对象结构。
- 必须保留 `schema_version`。
- 应用层要做 JSON Schema 或结构校验，不能裸透传。

### 9.3 第三方回包类

- `cdd_merchant_mini_program.last_detect_result_json`
- `cdd_order_pay_record.pay_request_json`
- `cdd_order_pay_record.pay_response_json`

规则：
- 原始报文与业务摘要分开处理。
- 高频排查字段要冗余为显式列，例如第三方流水号、检测结果总状态。

## 10. 数据生命周期与清理策略

| 表类型 | 清理建议 |
| --- | --- |
| 购物车表 | 超过 180 天未更新的数据可归档或清理 |
| 结算快照表 | 超过 30 天且未转单的数据可清理 |
| 发布日志表 | 明细日志保留 180 天，摘要长期保留 |
| 审计日志表 | 至少保留 365 天 |
| 支付/退款流水 | 不做物理删除，长期保留 |
| 订单状态日志 | 至少保留到订单归档周期结束 |
| 报表日汇总 | 长期保留，必要时按年月归档 |

补充规则：
- 软删除不等于归档。
- 清理脚本必须按时间批量执行，避免大事务。
- 生产环境删除类任务必须保留执行审计。

## 11. 从草案到正式 DDL 的收口顺序

### 11.1 先冻结以下统一规则

1. 状态字典取值。
2. 金额字段命名。
3. 主键与业务单号策略。
4. 租户字段必带规则。
5. JSON 字段允许范围。

### 11.2 再输出正式迁移脚本

建议按下面顺序拆迁移文件：

1. `V1__auth_and_config.sql`
2. `V2__merchant.sql`
3. `V3__decoration.sql`
4. `V4__product.sql`
5. `V5__order.sql`
6. `V6__release.sql`
7. `V7__marketing_and_report.sql`
8. `V8__seed_data.sql`

当前已落盘到：`db/migration/`

### 11.3 种子数据建议

- `cdd_config_id_segment` 初始化业务号段。
- `cdd_auth_role` 初始化平台管理员、交付管理员、商家主账号角色。
- `cdd_auth_permission` 初始化菜单与动作权限点。
- `cdd_config_feature_switch` 初始化一期功能开关。
- `cdd_template_version` 初始化首个默认模板版本。
