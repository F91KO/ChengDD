# 前端交付与验收说明

## 目标
- 把 `cdd-frontend` 作为一期后端完成后的补充收口，确保其安装、运行、构建、验收流程可追溯。
- 把前端测试结果输出到 `docs/05-delivery/reports/frontend-acceptance.md`，并作为一期交付的额外验收材料。

## 验收流程
1. `corepack pnpm install`：确保依赖安装成功并写入 `pnpm-lock.yaml`。
2. `corepack pnpm build`：确认类型检查和构建都通过，避免编译错误。
3. `corepack pnpm dev --host 127.0.0.1 --port 4173` + `curl -I -L http://127.0.0.1:4173/`：验证开发服务器能启动并返回 `200 OK`。
4. 若后端接口可用，确保登录、当前身份、商品列表、订单列表、工作台报表至少有一次真实接口验证。
5. 若接口返回空数组，也应视为“链路通、业务数据未就绪”，报告里要明确区分，不要误写成接口失败。
6. 若某个服务端口被占用但接口仍不可访问，应在报告里明确记录为“本地服务异常/残留进程”，不要写成已验通过。

## 交付产物
- 更新 `docs/05-delivery/当前任务收口清单.md`，把前端补充收口写入当前一次迭代状态。
- 将测试细节与风险记录在 `docs/05-delivery/reports/frontend-acceptance.md`。
- 如果后续需要新增页面或真实接口，还需要：
  - 更新 `cdd-frontend/src` 中的组件/页面文案 & 状态。
  - 回滚/再执行上述验收流程并更新报告。

## 角色分工
- 主 agent：负责问题拆解、issue、PR、验收结论、与后端收口一致的汇总。
- 文档 agent：负责这份说明文档、测试报告和说明 `cdd-frontend` 的补充角色。
- 测试 agent：负责记录执行命令、成功/失败、是否达标，以及未覆盖接口的风险项。

## 备注
- 这个文档属于“交付与商务”类，既记录执行步骤也方便交付检查。只要后台问题未重新开启，后端依旧视为“已完成一期基线”。
- 当前前端不再允许保留假数据回退或内存假数据；若接口未接通，应直接记录为真实链路未完成，并输出风险说明。
- 当前前端文案、状态提示、验收结论统一使用中文。
- 当前工作台页面已接入 `report-service` 的 `/api/report/*` 真实接口；若后续增加网关聚合层，需要同步调整代理、页面请求地址与验收报告。
