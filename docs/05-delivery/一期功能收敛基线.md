# 一期功能收敛基线

## 1. 目标

- 目的：为一期交付建立统一范围基线，避免继续以零散 issue、实现现状或临时讨论扩 scope。
- 判断标准：以 [PRD 4.1 本期范围](../01-product/ecommerce-marketplace-prd.md) 与 [PRD 8.7 交付范围](../01-product/ecommerce-marketplace-prd.md) 为业务基线，以当前交付文档为落地口径。
- 一期交付目标收敛为：能开通、能发布、能看数据。

## 2. 一期必须交付的业务闭环

### 2.1 平台与商家接入

- 平台账号登录、审核商家入驻申请。
- 商家主体资料、店铺资料、小程序接入与激活。
- 主账号 / 子账号权限与数据范围控制。

### 2.2 商品与模板基线

- 平台维护分类模板，商家可基于平台模板初始化自己的分类树。
- 商家可维护商品分类、商品、SKU、库存等基础目录能力。
- 分类属性模板保留为平台提供的初始化来源。

### 2.3 交易闭环

- 消费者完成购物车、下单、支付、退款 / 售后基础链路。
- 平台和商家可查询订单结果、退款结果和关键状态日志。

### 2.4 发布与配置

- 模板版本、发布任务、结果同步、回滚等基础发布能力。
- 平台配置、商家覆盖、功能开关等基础配置能力。
- 装修基础配置可支撑首页模板初始化与首版发布。

### 2.5 数据查看

- 商家和平台至少具备基础看板、订单 / 商品结果查询能力。
- 一期验收关注“有真实数据可看”，不要求一次性扩展完整报表体系。

## 3. 一期明确保留的边界

- 分类模板管理归属平台后台，商家侧只保留“查询可用模板并初始化分类树”的使用链路。
- 商家后台核心页面在一期范围内必须走真实接口，不接受 mock 或内存假数据作为验收结果。
- 分类属性模板在一期只要求满足初始化来源语义，不要求展开复杂治理。
- 前端控制台属于交付补充物，用于承接真实接口联调与演示；是否存在补充页面，不改变一期业务范围定义。

## 4. 明确后置项

- 复杂分类属性模板管理。
- 模板审批、复杂版本治理、批量治理等扩展流程。
- 订单发货模型纠偏。
- 订单状态契约纠偏。
- 超出“能开通、能发布、能看数据”目标的报表、运营分析和平台治理扩展。

## 5. 不纳入一期业务验收判定的工程支撑项

- Agent Core 等研发辅助能力。
- 自动化测试脚本、骨架校验脚本、交付脚本本身。
- 本地 Docker、Nacos、MySQL 启动流程等研发环境能力。

说明：
- 上述内容仍然需要维护和验收，但它们属于“一期工程交付支撑”，不单独决定一期业务范围是否继续扩张。

## 6. 当前结论

- 一期业务基线以“核心闭环可用”为准，不再继续吸纳新的复杂治理能力。
- 已确认仍存在的订单纠偏项保留在后续 backlog 中推进，但不再与“一期范围是否继续扩大”混在一起讨论。
- 后续若新增需求，应新开 issue、分支和验收口径，不回写为一期必做项。
