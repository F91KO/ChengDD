# 需求漂移纠偏 Issue 草案

## 使用说明

- 本文用于后续创建 GitHub issue。
- 创建顺序遵循：
  1. 先确认任务 list
  2. 再创建 issue
  3. 再创建分支
- 当前不直接建 issue，仅输出草案。
- 当前优先级规则：
  - `P0`：分类、分类模板、商品
  - `P1`：订单发货模型纠偏、订单状态契约纠偏

## Issue P0-1

### 标题

`P0 收口：商品分类页面真实联调与可用性修复`

### 建议分支

`feature-category-page-stabilization`

### 目标

- 保证商家侧商品分类页面在真实接口下可稳定使用。
- 清理影响分类页使用的明显 bug、死链、状态不一致和交互缺陷。

### 范围

- 分类列表加载、分页、搜索、选择态
- 平台分类模板初始化分类树
- 新增一级分类 / 新增子分类
- 编辑分类
- 启用 / 停用 / 展示 / 隐藏状态操作
- 页面空态、失败态、操作提示与刷新逻辑

### 不在范围

- 本轮不建设新的分类批量治理能力
- 本轮不扩展复杂分类属性模板管理

### 交付物

- 商品分类页面真实接口联调收口
- 核心操作问题清单及修复结果
- 本地验证记录与验收结论

### 验收标准

- 商品分类页核心操作均走真实接口
- 初始化后能立即查看并继续编辑分类树
- 分页、搜索、选择、编辑之间无明显状态污染
- 无按钮无响应、成功失败提示缺失、提交后不刷新的明显问题

### 主要模块

- `cdd-frontend`
- `cdd-parent/cdd-api-product`
- `cdd-parent/cdd-product-service`
- `docs/05-delivery`

### 依赖

- 依赖当前真实商品域接口可访问

## Issue P1-1

### 标题

`P1 纠偏：订单发货恢复物流发货模型`

### 建议分支

`feature-order-shipping-alignment`

### 目标

- 将当前“履约状态推进”能力纠正为需求定义的“物流发货”能力。
- 保证发货接口、请求字段、状态校验和前端交互全部与需求一致。

### 范围

- 前端发货交互改为物流发货表单
- 前端请求地址与字段恢复需求口径
- 后端请求模型补齐物流字段
- Controller、应用服务、仓储校验恢复物流发货语义
- 联调与验收文档同步调整

### 不在范围

- 本轮不扩展第三方物流轨迹能力
- 本轮不接入物流公司编码字典平台化管理
- 本轮不引入复杂履约拆单场景

### 交付物

- 前端物流发货表单
- 后端物流发货接口与校验
- 本地联调记录与验收结论

### 验收标准

- 未填写物流公司编码、物流公司名称、运单号时不能发货
- 仅允许需求规定状态的订单执行发货
- 发货后订单详情或列表可查询到完整物流信息
- 前端、接口、文档使用相同发货语义

### 主要模块

- `cdd-frontend`
- `cdd-parent/cdd-api-order`
- `cdd-parent/cdd-order-service`
- `docs/03-interface`
- `docs/05-delivery`

### 依赖

- 依赖需求基线冻结
- 建议与订单状态契约纠偏连续推进，避免再次分叉

## Issue P1-2

### 标题

`P1 纠偏：订单状态契约恢复需求口径`

### 建议分支

`feature-order-status-contract-alignment`

### 目标

- 统一前端、接口、文档、验收中的订单状态表达，恢复到需求文档定义口径。
- 消除当前 `order_status + delivery_status` 对外表达与需求不一致的问题。

### 范围

- 梳理需求状态枚举和流转规则
- 识别当前代码中的状态拆分点和映射关系
- 统一接口返回与页面展示状态
- 修正接口文档、前端文案、验收文档

### 不在范围

- 本轮不新增更复杂的订单生命周期节点
- 本轮不重构所有内部状态字段存储结构，除非为满足对外契约必须调整

### 交付物

- 订单状态契约对照表
- 对外状态口径统一后的接口与页面
- 同步更新的文档与验收说明

### 验收标准

- 订单列表、订单详情、接口响应中的状态名称与需求一致
- 文档中不再保留与当前基线冲突的旧状态说明
- 如内部保留细粒度状态，必须存在明确映射规则且不对外泄漏歧义

### 主要模块

- `cdd-frontend`
- `cdd-parent/cdd-api-order`
- `cdd-parent/cdd-order-service`
- `docs/03-interface`
- `docs/05-delivery`

### 依赖

- 依赖需求基线冻结
- 与订单发货模型纠偏强相关，建议同一批次推进

## 推荐创建顺序

1. `P0 收口：商品分类页面真实联调与可用性修复`
2. `P0 收口：分类模板归位平台后台并打通商家初始化链路`
3. `P0 收口：商品管理页面真实联调与可用性修复`
4. `P1 纠偏：订单发货恢复物流发货模型`
5. `P1 纠偏：订单状态契约恢复需求口径`

## 推荐执行顺序

1. 先做商品分类页面收口
2. 再做分类模板归位与商家初始化链路收口
3. 再做商品管理页面收口
4. 最后做订单发货模型与订单状态契约纠偏
