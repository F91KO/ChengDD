# 生产环境 K8s 多商户部署建议

## 1. 文档目标

- 明确程哒哒在生产环境使用 Kubernetes 承载多商户业务时的推荐部署方式。
- 统一“多商户共享平台”和“单商户独立部署”的边界，避免后续架构失控。
- 作为后续环境建设、交付方案、容量规划和运维策略的基线文档。

## 2. 核心结论

- 生产环境可以使用一套 `K8s` 集群服务多个商户。
- 默认推荐模式是“统一平台 + 多租户隔离”，不是“每个商户一套独立系统”。
- 应用服务可以部署在 `K8s` 中，生产数据库不建议采用“单 Pod 普通容器”的轻量方式托管。
- `Nacos` 可以部署在 `K8s` 中，但生产应按高可用要求建设，不建议直接照搬本地 `standalone` 模式。

## 3. 推荐的总体模式

### 3.1 默认模式

- 一套生产集群
- 一套统一代码主干
- 一组共享业务服务实例
- 多个商户共用平台能力
- 通过数据、权限、配置、模板做商户隔离

这也是一期最符合当前产品方向和交付成本的模式。

### 3.2 不推荐作为默认模式

- 每个商户单独一套 `Deployment`
- 每个商户单独一套 `Namespace`
- 每个商户单独维护长期代码分支
- 每个商户绑定一套独立数据库容器

这种模式会快速放大：

- 交付成本
- 发布复杂度
- 配置维护成本
- 故障排查复杂度
- 版本收敛难度

## 4. 多商户隔离分层建议

### 4.1 一期推荐隔离等级

- 应用层：共享服务实例
- 数据层：共享库共享表，按 `merchant_id` 等租户字段隔离
- 配置层：按商户配置、功能开关、模板版本隔离
- 权限层：按账号、角色、权限点隔离

这是标准 SaaS 多租户模式，也是当前最现实的落地方案。

### 4.2 二期可升级隔离

对于重点商户、监管要求高的商户，可以逐步提升隔离等级：

- 共享集群，独立数据库
- 共享集群，独立命名空间
- 共享代码，独立配置集
- 特定商户独立发布批次

### 4.3 仅在特殊场景考虑独占部署

仅当满足以下情况之一时，再考虑“单商户独立部署”：

- 大客户强隔离要求
- 合规审计要求独立资源边界
- 单商户业务体量足够大
- 客户采购的是独立私有化版本

## 5. K8s 中建议承载的组件

适合部署到 `K8s`：

- `cdd-gateway`
- `cdd-auth-service`
- `cdd-merchant-service`
- `cdd-decoration-service`
- `cdd-product-service`
- `cdd-order-service`
- `cdd-marketing-service`
- `cdd-release-service`
- `cdd-report-service`
- `cdd-config-service`
- 业务异步任务消费者
- 日志、监控、告警代理

## 6. 数据库与中间件建议

### 6.1 MySQL

生产环境不建议直接采用“业务项目里的 `docker compose` 或单容器 MySQL”方式部署。

更推荐：

- 云数据库：如 `RDS`
- 独立自建高可用 MySQL 集群

原因：

- 备份恢复更规范
- 主从与故障切换更成熟
- 监控、审计、权限治理更完整
- 运维职责与业务容器解耦

### 6.2 Redis

推荐：

- 云 Redis
- 或独立高可用 Redis 集群

### 6.3 Nacos

可以放在 `K8s` 中，但应按生产标准建设：

- 非本地 `standalone`
- 持久化存储
- 集群模式
- 明确 namespace、group、dataId 规范
- 开启鉴权与访问控制
- 做配置备份与变更审计

### 6.4 MQ

如果后续补齐异步解耦与补偿执行器协同，建议 `MQ` 也作为平台级共享基础设施，不按商户单独部署。

## 7. 推荐的环境隔离方式

推荐按“环境”隔离，而不是按“商户”隔离：

- `dev`
- `test`
- `prod`

在 `prod` 集群内再按系统职责划分资源边界，而不是按商户切割整套系统。

示例：

- `chengdd-gateway`
- `chengdd-core-services`
- `chengdd-observability`
- `chengdd-middlewares`

## 8. K8s 资源组织建议

### 8.1 Namespace

推荐按环境或平台职责拆分：

- `chengdd-prod-gateway`
- `chengdd-prod-app`
- `chengdd-prod-ops`

不建议默认按商户拆 `namespace`。

### 8.2 Ingress

- 统一入口
- 支持多域名、多证书
- 支持灰度发布与流量切换

### 8.3 ConfigMap / Secret

- 非敏感配置放 `ConfigMap`
- 密钥、数据库密码、令牌密钥放 `Secret`
- Nacos 接入后，K8s 侧仅保留最小启动与基础连接参数

### 8.4 HPA

- 网关、认证服务、核心读流量服务建议配置弹性伸缩

### 8.5 ResourceQuota 与 LimitRange

- 避免个别服务异常占满节点资源

### 8.6 NetworkPolicy

- 明确服务之间访问边界
- 中间件访问范围白名单化

## 9. 应用发布建议

- 所有商户基于统一平台版本发布
- 商户差异优先通过：
  - 配置
  - 功能开关
  - 模板
  - 权限
  - 数据规则
- 不为单商户长期维护独立代码主线

重点客户如确需差异发布，应控制为：

- 短周期分支
- 临时灰度
- 明确回收计划

## 10. 数据隔离建议

一期建议：

- 所有核心业务表都保留租户维度字段
- 查询、更新、统计都必须显式带商户边界
- 网关与认证上下文负责注入租户信息
- 服务内部禁止前端随意越权指定其他商户数据

后续如需增强：

- 可逐步升级到独立 schema 或独立数据库
- 但不建议一开始就直接按商户拆散

## 11. 适合当前项目的推荐拓扑

推荐拓扑：

- `K8s` 承载全部业务应用
- `MySQL` 使用托管数据库或独立数据库集群
- `Redis` 使用托管或独立高可用集群
- `Nacos` 作为平台级配置中心
- `Ingress + HPA + 监控告警 + 日志采集` 作为标准平台能力

简单理解：

- 应用上云上集群
- 数据库独立治理
- 商户共享平台
- 通过多租户机制做隔离

## 12. 当前阶段建议落地顺序

1. 本地完成 `MySQL + Nacos` 基础设施联调
2. 完成 `auth-service` 数据库持久化
3. 明确 `merchant_id` 级数据隔离链路
4. 补全配置中心接入方案
5. 再设计生产环境 `K8s` 拓扑、资源规格、发布流程

## 13. 最终建议

- 可以在生产环境搭建一套 `K8s` 给多个商户使用。
- 但应采用“多租户共享平台”模式，而不是“每个商户一套独立应用”模式。
- 生产数据库不要照搬本地 Docker 容器思路，优先使用托管数据库或正式高可用集群。
- `K8s` 的价值主要在于承载应用、流量治理、弹性扩缩、运维标准化，不应把所有状态型中间件都用最轻量方式塞进去。
