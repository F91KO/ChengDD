# #31 商品明细退货退款任务清单

## 1. 目标

- 在现有订单级退款基线之上，补齐“按商品明细退货退款”能力。
- 本轮按最小可落地方案实现：
  - 一个售后单对应一个订单项。
  - 支持 `refund_only` 与 `return_refund` 两类售后。
  - 售后单承接业务状态流转，退款单承接第三方退款执行与回调幂等。
- 不在本轮实现：
  - 一个售后单覆盖多个订单项。
  - 换货流程。
  - 复杂逆向物流履约。

## 2. 统一约束

- 统一售后状态枚举，采用文档已定义的业务语义：
  - `pending_merchant`
  - `agreed`
  - `rejected`
  - `waiting_return`
  - `refunding`
  - `completed`
  - `closed`
- 统一售后类型枚举：
  - `refund_only`
  - `return_refund`
- 订单维度 `pay_status` 继续保留：
  - `paid`
  - `refund_partial`
  - `refund_full`
- 订单项退款状态不再整单同步，必须按目标 `order_item` 更新。
- 金额与数量校验同时满足：
  - 订单项累计退款金额不能超过该项可退金额。
  - 订单项累计退款数量不能超过购买数量。
  - 订单累计成功退款金额不能超过订单实付金额。

## 3. 设计落点

### 3.1 数据模型

- 扩展 `cdd_order_after_sale`，作为商品明细售后主单。
- 扩展 `cdd_order_refund_record`，关联售后单与订单项。
- 扩展 `cdd_order_item`，维护 item 级累计退款金额与数量。
- 保留 `cdd_order_refund_callback_record`，继续承担退款回调幂等。

### 3.2 接口边界

- 不继续复用“整单退款”语义承接退货退款。
- 新增 `/after-sales` 相关接口，售后与退款分层：
  - 发起售后申请
  - 商家审核售后
  - 用户提交退货物流
  - 第三方退款回调
- 原订单级退款接口保留，后续可作为兼容入口或仅退款快捷入口评估是否收敛。

## 4. 任务拆解

### T1 统一售后术语与状态

- 对齐文档、接口、数据库设计中的售后状态枚举。
- 明确本轮采用“一个售后单对应一个订单项”的边界。
- 输出代码中的常量、字段命名和响应值约束。

验收：
- 文档、接口模型、数据库脚本中的售后状态不再出现 `applied/reviewing/approved/finished` 这套旧值。

### T2 数据库迁移与测试 Schema

- 新增正式迁移脚本，建议命名：
  - `db/migration/V17__order_item_level_after_sale_refund.sql`
- 扩展 `cdd_order_after_sale` 字段，至少包含：
  - `after_sale_no`
  - `after_sale_type`
  - `after_sale_status`
  - `order_item_id`
  - `refund_record_id`
  - `refund_quantity`
  - `refund_amount`
  - `reason_code`
  - `reason_desc`
  - `proof_urls_json`
  - `merchant_result`
  - `return_company`
  - `return_logistics_no`
  - `returned_at`
  - `approved_at`
  - `completed_at`
  - `closed_at`
- 扩展 `cdd_order_refund_record` 字段，至少包含：
  - `after_sale_id`
  - `order_item_id`
  - `refund_quantity`
- 扩展 `cdd_order_item` 字段，至少包含：
  - `refunded_quantity`
  - `refunded_amount`
- 补充必要索引：
  - `cdd_order_after_sale(order_item_id, after_sale_status)`
  - `cdd_order_refund_record(after_sale_id)`
  - `cdd_order_refund_record(order_item_id, refund_status)`
- 同步更新 H2 `schema.sql` 与本地/测试数据。

验收：
- MySQL 迁移幂等执行通过。
- H2 测试 schema 与正式迁移字段一致。

### T3 API 模型与 Controller

- 在 `cdd-api-order` 新增或调整售后模型：
  - `OrderAfterSaleCreateRequest`
  - `OrderAfterSaleReviewRequest`
  - `OrderAfterSaleReturnRequest`
  - `OrderAfterSaleLifecycleResponse`
  - 必要时新增 item 明细请求/响应模型
- 在 `OrderItemResponse` 增加 item 级退款展示字段。
- 在 `OrderController` 新增 `/after-sales` 路由。

验收：
- 创建、审核、退货录入、查询响应可表达 item 级售后与退款信息。

### T4 仓储与应用服务

- 在 `OrderRepository` / `JdbcOrderRepository` 新增能力：
  - 查询指定订单项
  - 查询订单项累计成功/处理中退款金额与数量
  - 创建售后单
  - 更新售后状态
  - 仅更新目标订单项的退款状态、累计退款金额、累计退款数量
  - 按订单汇总累计退款金额
- 在 `OrderApplicationService` 实现链路：
  - 发起售后申请
  - 商家同意 / 拒绝
  - 退货退款场景提交物流后进入退款中
  - 创建退款执行单
  - 退款回调成功 / 失败
  - 回写订单项状态与订单支付状态
- 保留现有退款补偿任务机制，失败回调继续绑定补偿任务。

验收：
- 任一订单项退款不会再把整单所有 `order_item.refund_status` 一起覆盖。
- 订单 `pay_status` 根据 item 级累计成功退款额正确回写。

### T5 错误码与校验

- 新增细粒度错误码，至少包含：
  - `ORDER_ITEM_NOT_FOUND`
  - `AFTER_SALE_NOT_FOUND`
  - `AFTER_SALE_STATUS_INVALID`
  - `AFTER_SALE_TYPE_INVALID`
  - `REFUND_ITEM_AMOUNT_INVALID`
  - `REFUND_ITEM_QUANTITY_INVALID`
  - `RETURN_LOGISTICS_REQUIRED`
- 补充参数与业务校验：
  - 非本订单项不可申请售后
  - 重复退款金额超限
  - 退款数量超限
  - `return_refund` 未审核通过前不可提交退货物流
  - `refund_only` 不允许提交退货物流

验收：
- 非法状态流转与超限退款请求能返回明确业务错误。

### T6 自动化测试与本地验证

- 补充 `cdd-order-service` 集成测试，至少覆盖：
  - 单个订单项仅退款成功
  - 单个订单项退货退款成功
  - 商家拒绝售后
  - 同一订单项累计部分退款控制
  - 两个订单项分别退款，互不污染
  - 重复退款回调幂等
  - 退款失败补偿任务绑定
- 本地验证动作：
  - `mvn -q -f cdd-parent/pom.xml -pl cdd-order-service -am test`
  - 本地启动 `cdd-order-service`
  - 关键接口手工验证或脚本验证
- 输出测试报告并更新交付文档。

验收：
- 自动化测试全部通过。
- 本地服务启动成功，核心售后接口可用。

## 5. 开发顺序

1. 先做 T1、T2，锁定状态与表结构。
2. 再做 T3、T4，补 API 和核心业务。
3. 最后做 T5、T6，补校验、测试、报告和文档。

## 7. 备注

- 现有订单级退款能力不应直接删除，避免影响一期基线。
- 若后续确认需要“一次售后覆盖多个订单项”，再单独新增 `after_sale_item` 明细表并另开迭代。
