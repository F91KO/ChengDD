# 权限建议清单

## 1. 文档说明

- 文档用途：整理后续在 `/permissions` 中建议配置的持久化命令前缀。
- 适用范围：当前仓库的 Git、GitHub 和本地验收执行流程。
- 说明：该清单是人工配置参考，不会被 `/permissions` 自动从仓库文件导入。

## 2. 使用原则

- 优先配置稳定、可复用、范围适中的命令前缀。
- 优先覆盖高频动作，减少重复授权确认。
- 不把整条长命令原样当作规则维护，优先提炼前缀。
- 破坏性命令不加入持久化授权。

## 3. 建议持久化前缀

建议优先加入以下前缀：

```text
git add
git commit -m
git switch -c
git push
gh issue list
gh issue create
gh issue close
gh pr create
gh pr view
gh pr edit
bash scripts/validation/validate_backend_skeleton.sh
```

## 4. 适用场景

- `git add`
  - 暂存当前任务改动
- `git commit -m`
  - 按 issue 范围生成提交
- `git switch -c`
  - 为新任务创建分支
- `git push`
  - 推送任务分支
- `gh issue list`
  - 查询 issue 状态
- `gh issue create`
  - 创建新任务 issue
- `gh issue close`
  - 关闭已完成 issue
- `gh pr create`
  - 创建 PR
- `gh pr view`
  - 查询 PR 状态或链接
- `gh pr edit`
  - 更新 PR 标题、描述、关联 issue
- `bash scripts/validation/validate_backend_skeleton.sh`
  - 执行当前统一骨架验收

## 5. 不建议持久化前缀

以下命令不建议加入持久化授权：

```text
rm
git reset --hard
git checkout --
```

原因：

- 这些命令具备明显破坏性，容易误删文件或回滚用户未保存的改动。

## 6. 配置建议

- 在客户端输入 `/permissions`
- 逐条添加上面的建议前缀
- 选择“持久允许”或等价选项
- 后续新增命令时，优先补前缀规则而不是只放开单次完整命令

## 7. 当前项目约束

- 后续执行 GitHub 和 Git 操作时，先检查是否已有可复用授权。
- 若没有，再申请新的前缀授权。
- 任务执行流程仍遵循：先任务 list，再 issue，再分支，再实现与 PR。
